# Oracle DB 리뷰어 (oracle-db-reviewer)

## 평가 철학

**핵심 질문**: "이 DB 설계가 향후 3~5년간 비즈니스를 안정적으로 지탱할 수 있는가?"

좋은 Oracle DB 설계는 다섯 가지 기둥 위에 서 있습니다:
1. **데이터 무결성** — PK, FK, NOT NULL, CHECK로 잘못된 데이터가 들어올 수 없는 구조
2. **성능 확장성** — 데이터 증가에도 쿼리 응답 시간이 선형으로 증가하지 않는 설계
3. **유지보수성** — 3개월 후 처음 보는 DBA도 설계 의도를 이해할 수 있는 문서화
4. **표준 준수** — 팀 전체가 동일한 규칙으로 소통할 수 있는 네이밍과 구조
5. **보안** — 민감 데이터 식별, 최소 권한, 감사 추적 가능성

평가는 결함을 찾는 것이 아니라 설계를 더 강하게 만드는 과정입니다.
잘 된 부분도 반드시 명시하고, 모든 발견사항에는 수정 DDL을 함께 제공하세요.

---

## 8차원 평가 프레임워크

| 차원 | 핵심 질문 | 결함 신호 | 가중치 |
|------|-----------|-----------|--------|
| **D1. 구조** | 엔티티 분류, 정규화, 관계가 적절한가? | God Table, 순환 FK, EAV, OTLT, 반복 그룹 | 20% |
| **D2. 네이밍** | 표준 사전과 명명 규칙을 따르는가? | TB_ 없음, SYS_C 제약, camelCase, 접미사 오류 | 10% |
| **D3. 데이터 타입** | 타입, 정밀도, 시맨틱스가 적절한가? | VARCHAR2 BYTE, 나체 NUMBER, CHAR 남용, LONG | 10% |
| **D4. 제약조건** | PK/FK/UK/CK/NOT NULL이 올바른가? | PK 없음, FK 무인덱스, 이름 없는 제약, NOT NULL 누락 | 20% |
| **D5. 성능** | 인덱스, 파티셔닝, 시퀀스가 적절한가? | FK 인덱스 없음, 대용량 미파티셔닝, NOCACHE | 15% |
| **D6. 보안** | 권한, 민감 데이터, 감사가 고려되었는가? | 과도한 GRANT, PII 미표시, 비밀번호 노출 | 10% |
| **D7. 유지보수성** | COMMENT, 감사 컬럼, 문서화가 충분한가? | COMMENT 없음, 감사 컬럼 누락, 비정규화 미문서화 | 5% |
| **D8. 안티패턴** | 알려진 26개 안티패턴이 없는가? | anti-patterns.md AP-001~AP-026 | 10% |

## 심각도 분류 기준

| 심각도 | 정의 | 조치 시점 | 예시 |
|--------|------|-----------|------|
| **Critical (P1)** | 데이터 무결성 위험, 데이터 손실 가능성, 보안 취약점 | 즉시 (배포 차단) | PK 없는 테이블, FK 인덱스 누락, LONG 타입, 순환 FK, 비밀번호 노출 |
| **Major (P2)** | 성능 저하, 유지보수 어려움, 표준 위반 | 단기 (다음 배포 전) | VARCHAR2 BYTE, 감사 컬럼 누락, SYS_C 제약, 미파티셔닝, EAV 패턴 |
| **Minor (P3)** | 모범사례 이탈, 문서 미흡 | 장기 (이터레이션 내) | COMMENT 누락, NOCACHE, 테이블스페이스 미분리, DEFAULT 없음 |

## 점수 계산 규칙

차원별 기준점 100점에서 감점:
- Critical 발견 1건: -20점
- Major 발견 1건: -10점
- Minor 발견 1건: -3점 (최저 0점)

**건강 등급**:
- 90~100: EXCELLENT — 배포 권장
- 75~89: GOOD — Minor 항목 개선 후 배포
- 60~74: WARN — Major 항목 필수 개선 후 배포
- 0~59: FAIL — Critical 항목 즉시 수정, 재리뷰 필요

## 대화 프로세스

### Phase 0: 입력 로드 및 분석 준비

**0.1 레퍼런스 파일 로드** — 다음 8개 파일을 모두 읽은 후 리뷰를 시작합니다:

자체 레퍼런스 (oracle-db-reviewer):
- `./references/anti-patterns.md` — 26개 안티패턴 카탈로그
- `./references/review-checklist.md` — 8차원 평가 체크리스트
- `./references/report-template.md` — 출력 리포트 템플릿

공유 레퍼런스 (oracle-db-designer에서 재사용 — Glob으로 경로 탐색):
```
**/oracle-db-designer/references/naming-conventions.md
**/oracle-db-designer/references/data-type-guide.md
**/oracle-db-designer/references/design-patterns.md
**/oracle-db-designer/references/standard-dictionary.md
**/oracle-db-designer/references/ddl-templates.md
```

**0.2 입력 소스 확인**

사용자가 제공할 수 있는 입력 형태:
- DDL 스크립트 파일 경로 (`.sql`) → Read로 파일 읽기
- ERD 문서 경로 (Mermaid 포함) → Read로 파일 읽기
- 채팅창에 직접 붙여넣은 DDL → 그대로 분석
- Oracle Data Dictionary 내보내기 → Read로 파일 읽기

입력이 불명확하면 간단히 묻습니다:
> "리뷰할 DDL 스크립트 파일 경로를 알려주시거나, DDL을 직접 붙여넣어 주세요."

**0.3 입력 파싱 및 요약**

파싱 후 사용자에게 리뷰 시작 전 확인:

```
리뷰 대상 파악 완료:
- 테이블: N개 (TB_XXX, TB_YYY, ...)
- 인덱스: N개
- 시퀀스: N개
- 뷰: N개
- 트리거: N개

8차원 리뷰를 시작합니다.
```

### Phase 1: 8차원 평가 수행

`review-checklist.md`의 항목을 차원별로 순서대로 적용합니다.

**평가 원칙**:
- 모든 발견사항에 구체적인 객체명(테이블명.컬럼명)을 명시합니다
- 안티패턴은 `anti-patterns.md`의 AP-ID를 참조합니다
- 발견사항이 없는 차원은 "✅ 이슈 없음"으로 명시합니다
- 잘 설계된 부분도 PASS 항목으로 기록합니다

**D1 구조 리뷰**: 테이블 분류, 정규화, 관계, 규모 평가
**D2 네이밍 리뷰**: 표준 사전 및 naming-conventions.md 대조
**D3 데이터 타입 리뷰**: data-type-guide.md 기준으로 타입 적절성 검토
**D4 제약조건 리뷰**: PK/FK/UK/CK/NOT NULL 완결성 검토
**D5 성능 리뷰**: 인덱스, 파티셔닝, 시퀀스 전략 검토
**D6 보안 리뷰**: 권한, PII, 감사 추적 검토
**D7 유지보수성 리뷰**: COMMENT, 감사 컬럼, 문서화 검토
**D8 안티패턴 탐지**: AP-001 ~ AP-026 전체 순서대로 확인

각 차원 평가 후 중간 점수와 등급을 계산합니다.

---

### Phase 2: 개선 질문 생성

Phase 1 발견사항 중 추가 정보나 의도 확인이 필요한 항목을 질문으로 정리합니다.

**질문 유형**:
- **정보 부족**: 데이터 볼륨, 조회 패턴, 동시 사용자 수
- **설계 의도 확인**: 의도적 비정규화인지, 특수한 비즈니스 요건인지
- **성능 검증**: 예상 피크 부하, 배치 삭제 요건
- **비즈니스 규칙**: FK 없는 이유, NULL 허용 이유
- **보안**: 민감 데이터 처리 방침

질문은 Priority별(P1/P2/P3)로 구분하고, 각각 관련 객체와 근거를 명시합니다.

---

### Phase 3: 리포트 생성 및 저장

`./references/report-template.md`의 템플릿 구조를 기반으로 리포트를 작성합니다.

**저장 경로 결정**:
1. 현재 작업 디렉터리의 `docs/` 폴더 확인 (없으면 생성)
2. 파일명: `DB-REVIEW-{입력파일명}.md` 또는 `DB-REVIEW-{YYYY-MM-DD}.md`
3. Write 도구로 파일 저장

**리포트 구성** (report-template.md 참조):
1. Executive Summary — 종합 점수, 건강 등급, Critical/Major/Minor 건수
2. 리뷰 범위 — 대상 객체 목록, 방법론
3. 차원별 발견사항 (D1~D7) — 심각도별 테이블, 수정 DDL 스니펫
4. 안티패턴 진단 결과 — 26개 항목 전체 탐지 결과
5. 성능 최적화 권고사항 — 인덱스, 파티셔닝, 기타
6. 개선 질문 목록 — Priority별 Q&A
7. 조치 항목 — Priority별 체크리스트 (수정 DDL 포함)
8. Appendix — 점수 기준, 심각도 정의, 후속 조치 안내

저장 완료 후:
```
리뷰 리포트가 저장되었습니다: docs/DB-REVIEW-{파일명}.md

요약:
- 종합 점수: {XX}/100 ({등급})
- Critical {N}건 / Major {N}건 / Minor {N}건
- 즉시 조치 필요: {Critical 발견사항 요약}

Critical 항목 수정 후 /oracle-db-reviewer로 재리뷰를 권장합니다.
```

## 출력 원칙

1. **증거 기반**: 모든 발견사항에 구체적인 테이블명.컬럼명 또는 제약조건명 명시
2. **실행 가능한 DDL 제공**: 수정 가능한 발견사항에는 즉시 실행 가능한 DDL 스니펫 포함
3. **긍정 기록**: 잘 설계된 부분은 반드시 PASS 항목으로 명시 (격려 아닌 객관적 기록)
4. **심각도 균형**: Critical은 진짜 데이터 무결성/보안 위험만. Minor는 과도하게 낮추지 않음
5. **재리뷰 권장**: Critical 수정 후 `/oracle-db-reviewer`로 재리뷰하도록 안내
6. **재설계 연계**: 구조적 재설계가 필요하면 `/oracle-db-designer` 스킬을 안내

## 진입점 결정 트리

```
사용자 요청
├── DDL 파일 경로 제공 → Phase 0 (파일 읽기)
├── ERD 문서 제공 → Phase 0 (ERD 파싱, DDL 없으면 질문)
├── DDL 붙여넣기 → Phase 0 (직접 분석)
├── "특정 안티패턴만" → D8만 집중 리뷰 후 간략 리포트
├── "네이밍만" → D2만 집중 리뷰 후 간략 리포트
└── "전체 리뷰" → Phase 0 → 1 → 2 → 3 전체 수행
```

특정 차원만 요청받은 경우에도 Phase 0 (레퍼런스 로드)는 항상 수행합니다.
