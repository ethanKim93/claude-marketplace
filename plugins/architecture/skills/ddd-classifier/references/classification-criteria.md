# DDD 서브도메인 분류 기준 참조 가이드

## 1. 분류 유형 정의

### Core Domain (핵심 도메인)
비즈니스의 경쟁 우위를 창출하는 핵심 영역. 여기에 최고의 인재와 자원을 투입해야 한다.

**특징:**
- 비즈니스가 이 영역에서 경쟁한다 (차별화 포인트)
- 외부에서 구매하거나 아웃소싱할 수 없다 (하면 경쟁 우위 상실)
- 복잡한 비즈니스 규칙이 지속적으로 진화한다
- 도메인 전문가의 깊은 참여가 필수다
- 잘못 설계하면 비즈니스 전체가 실패한다

**예시:** 이커머스의 추천 알고리즘, 보험사의 요율 산출 엔진, 물류사의 최적 배송 경로 계산

---

### Supporting Domain (지원 도메인)
핵심 도메인을 지원하지만 그 자체로는 경쟁 우위가 아닌 영역.

**특징:**
- 비즈니스 특성을 반영한 맞춤 로직이 필요하다
- 그러나 경쟁사도 유사하게 구현한다
- 내부 개발이 적합하지만 최고 인재를 배치할 필요는 없다
- 또는 맞춤형 COTS(상용 소프트웨어)로 대체 가능하다

**예시:** 직원 온보딩 시스템, 내부 보고서 생성기, 상품 카탈로그 관리

---

### Generic Domain (일반 도메인)
업계 표준으로 해결된 문제 영역. SaaS, 오픈소스, 외부 서비스로 대체 가능.

**특징:**
- 모든 회사가 동일하게 필요로 한다
- 검증된 상용 솔루션이 존재한다
- 직접 구축은 자원 낭비다
- 비즈니스 로직이 단순하고 변경이 드물다

**예시:** 인증/인가(Auth0, Cognito), 이메일 발송(SendGrid), 파일 저장(S3), 결제(Stripe), 로깅/모니터링

---

## 2. 분류 결정 흐름

```
이 도메인이 비즈니스 경쟁의 핵심인가?
├── YES → Core Domain
└── NO  →
    이 도메인이 우리 비즈니스에 특화된 로직을 필요로 하는가?
    ├── YES → Supporting Domain
    └── NO  → Generic Domain
```

---

## 3. 가중 점수 매트릭스

각 서브도메인에 대해 아래 4개 기준을 1~5점으로 채점. 합산 점수로 분류를 보조한다.

| 기준 | 1점 | 3점 | 5점 | 가중치 |
|------|-----|-----|-----|--------|
| **비즈니스 차별화** | 모든 회사가 동일 | 우리 방식이 있음 | 핵심 경쟁 우위 | 40% |
| **비즈니스 규칙 복잡도** | 단순 CRUD | 중간 복잡도 | 복잡한 도메인 로직 | 25% |
| **변경 빈도** | 거의 변경 없음 | 분기별 변경 | 비즈니스와 함께 진화 | 20% |
| **외부 솔루션 대체 가능성** | 즉시 대체 가능 | 커스터마이징 필요 | 대체 불가 | 15% |

**판정 기준:**
- 가중 합산 **4.0 이상** → Core Domain 강력 후보
- 가중 합산 **2.5 ~ 3.9** → Supporting Domain 후보
- 가중 합산 **2.4 이하** → Generic Domain 후보

> **주의**: 점수는 보조 도구일 뿐이다. 비즈니스 맥락과 전략적 의도가 최종 판단 기준이다.

## 4. 소스코드 분석 휴리스틱

소스코드를 분석할 때 각 분류 유형을 시사하는 코드 패턴:

### Core Domain 시사 패턴
- 복잡한 비즈니스 규칙 메서드 (조건문 중첩, 도메인 용어가 풍부한 변수명)
- `@DomainService`, `@Aggregate`, `@Entity` 등 DDD 어노테이션 집중
- 도메인 이벤트가 활발히 발생하는 영역
- 단위 테스트가 촘촘히 작성된 패키지
- 빈번한 커밋 이력 (git log로 확인)
- 도메인 전문가 용어가 코드에 직접 반영됨

### Supporting Domain 시사 패턴
- 다른 도메인을 위한 조회/집계 서비스
- 어느 정도의 비즈니스 로직이 있지만 외부 API 호환성에 종속됨
- `*Manager`, `*Handler`, `*Processor` 같은 네이밍
- 중간 복잡도의 비즈니스 규칙

### Generic Domain 시사 패턴
- 단순 CRUD 작업 위주
- 외부 라이브러리/SDK 래퍼 코드
- `*Config`, `*Util`, `*Helper` 위주
- 인증/인가, 이메일/SMS, 파일 업로드, 결제 게이트웨이 연동
- 비즈니스 로직 없는 데이터 변환 코드

## 5. 문서 분석 휴리스틱

BRD/PRD/설계서 등 문서에서 도메인을 추출할 때 주목할 요소:

### 도메인 경계 신호
- 문서에서 용어가 맥락에 따라 다른 의미로 쓰이는 부분 → 경계 후보
- "~팀이 담당", "~시스템에서 관리" 같은 소유권 표현 → 컨텍스트 경계
- 데이터 복사/동기화를 언급하는 부분 → 컨텍스트 간 경계

### Core Domain 시사 표현
- "핵심 경쟁력", "차별화 요소", "우리만의 방식"
- 상세한 비즈니스 규칙 명세 (조건, 예외, 특수 케이스)
- 임원/도메인 전문가가 직접 설명하는 섹션

### Supporting/Generic 시사 표현
- "기존 시스템 연동", "외부 서비스 사용"
- "표준 절차", "일반적인 방식으로"
- "~와 유사하게 구현"

## 6. 흔한 분류 실수

| 실수 | 잘못된 생각 | 올바른 판단 |
|------|------------|------------|
| **복잡도 = Core** | "코드가 복잡하니 Core Domain이다" | 복잡도는 Core의 필요조건이 아님. 아이덴티티 관리는 복잡해도 Generic |
| **과도한 Core 지정** | "다 중요하니 다 Core다" | Core는 1~3개여야 집중 투자 효과가 있음. 많으면 재검토 |
| **전략 무시** | "현재 코드 구조대로 분류한다" | 분류는 현재 상태가 아닌 비즈니스 전략 기반 |
| **영구 분류** | "한 번 Core면 계속 Core다" | 비즈니스 전략이 바뀌면 분류도 바뀜 (예: 결제가 Core인 핀테크) |
| **조직 구조 반영** | "팀이 나뉘어 있으니 별도 도메인" | 콘웨이의 법칙은 참고지, 분류 기준이 아님 |
| **인프라를 Core로** | "DB가 복잡하니 Core" | 인프라 복잡도와 도메인 분류는 무관 |

---

## 7. Core Domain 수 가이드라인

| 조직 규모 | 권장 Core Domain 수 | 이유 |
|-----------|--------------------|----- |
| 스타트업 (< 20명) | 1개 | 집중이 생존 |
| 중소기업 (20~200명) | 1~2개 | 선택과 집중 |
| 중견기업 (200~2000명) | 2~3개 | 다각화 가능 |
| 대기업 (> 2000명) | 3~5개 | 포트폴리오 전략 |

Core Domain이 5개 이상이라면, 실제로 Supporting인 것들을 Core로 오분류했을 가능성이 높다.

---

## 8. 구현 전략 매핑

| 분류 | 권장 구현 전략 | 팀 배치 | 기술 선택 기준 |
|------|--------------|---------|--------------|
| Core Domain | 사내 개발, 최고 인재 배치 | 시니어 엔지니어 + 도메인 전문가 | 유연성·확장성 최우선 |
| Supporting Domain | 사내 개발 또는 맞춤형 COTS | 일반 엔지니어 | 적정 품질, 비용 고려 |
| Generic Domain | SaaS/오픈소스 구매 또는 외주 | 최소한의 통합 인력 | 검증된 솔루션, 통합 용이성 |
