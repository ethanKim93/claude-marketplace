# DDD Classifier — DDD 도메인 분류 문서 생성기

소스코드 또는 기존 문서를 분석하여 **표준 DDD 도메인 분류 문서**를 생성한다. 입력 형태와 무관하게 항상 동일한 템플릿 포맷의 결과물을 만들어낸다.

## 기존 `/ddd` 스킬과의 차이

| 구분 | `/ddd` | `/ddd-classifier` (이 스킬) |
|------|--------|---------------------------|
| 목적 | 대화형 DDD 분석 가이드 | 표준 분류 문서 생성기 |
| 출력 | 단계별 분석 및 설계 결과 | 단일 종합 마크다운 문서 |
| 초점 | 전술적 설계 (Aggregate, 패키지 구조) | 전략적 설계 (서브도메인 분류, 컨텍스트 맵) |
| 연계 | — | 이 문서를 기반으로 `/ddd` 심층 분석 가능 |

---

## 전체 프로세스 개요

```
Phase 0: 입력 수집 및 정규화
    ├── 0-A: 소스코드 모드 (Glob/Grep/Read로 프로젝트 스캔)
    └── 0-B: 문서 모드 (기존 문서 Read 및 도메인 요소 추출)
         ↓ (두 모드 모두 동일한 '내부 추출 메모'로 수렴)
Phase 1: 서브도메인 식별 및 분류
Phase 2: 바운디드 컨텍스트 & 컨텍스트 맵
Phase 3: 도메인 상세 (용어집, 이벤트, 애그리게이트)
Phase 4: 문서 생성 및 저장
```

각 Phase 완료 후 사용자 확인을 받고 다음 Phase로 진행한다. 사용자가 분류를 수정하거나 정보를 추가하면 그대로 반영한다.

## Phase 0: 입력 수집 및 정규화

### 시작 질문

먼저 아래 내용을 확인한다:

1. **입력 유형**: 소스코드 분석인가, 기존 문서 기반 분석인가?
   - 소스코드가 있으면 → **0-A 소스코드 모드**
   - BRD/PRD/설계서/아키텍처 문서가 있으면 → **0-B 문서 모드**
   - 이미 도메인이 분류된 문서도 0-B로 처리 (내용 추출 후 동일 템플릿으로 재생성)
2. **프로젝트명**: 결과 문서 파일명에 사용
3. **저장 위치**: 문서를 저장할 경로 (기본값: `docs/DDD-Classification-{프로젝트명}.md`)

두 가지 이상을 한꺼번에 받아도 좋다 — 예를 들어 소스코드와 BRD가 둘 다 있으면 둘 다 활용한다.

---

### Phase 0-A: 소스코드 모드

소스코드 기반 분석 시 아래 순서로 스캔한다. 대형 프로젝트는 전부 읽지 말고 전략적으로 샘플링한다.

**Step 1 — 프로젝트 구조 파악**
```
Glob으로 디렉터리 트리 파악:
- src/ 또는 루트의 모듈/패키지 구조
- build.gradle, pom.xml, package.json, go.mod, pyproject.toml 등 빌드 파일
- README.md, CLAUDE.md 등 설명 파일
```

**Step 2 — 도메인 패턴 탐색**
```
Grep으로 핵심 패턴 검색:
- @Entity, @Aggregate, @AggregateRoot
- @DomainEvent, extends DomainEvent
- interface *Repository, implements *Repository
- @DomainService, @ApplicationService
- package/module 선언부 (도메인명 추출)
- 이벤트 클래스명 패턴: *Created, *Updated, *Placed, *Approved
```

**Step 3 — 핵심 파일 선별 읽기**
```
아래 기준으로 읽을 파일 선별 (너무 많으면 최대 20개):
- 패키지/모듈 최상위 인터페이스 또는 루트 엔티티
- 주요 서비스 클래스 (비즈니스 로직 집중 파일)
- DB 스키마 또는 마이그레이션 파일 (테이블명 = 도메인 힌트)
- API 라우팅/컨트롤러 파일 (엔드포인트 구조)
```

**Step 4 — 내부 추출 메모 작성**

분석이 끝나면 아래 형식으로 정리한다 (사용자에게 보여준다):

```
## 소스코드 분석 결과 요약

### 발견된 비즈니스 역량 후보
1. {역량명}: {발견 근거 — 파일/패키지명}
2. ...

### 서브도메인 후보
1. {이름} — {관련 파일/클래스} — 예상 분류: Core/Supporting/Generic
2. ...

### 발견된 도메인 용어
{용어 목록}

### 확인된 통합 지점 (외부 시스템, 이벤트)
{통합 지점 목록}

### 추가 확인 필요 사항
{소스코드에서 불분명한 비즈니스 맥락}
```

### Phase 0-B: 문서 모드

문서(BRD/PRD/설계서/아키텍처 문서/기존 도메인 분류 문서 등)를 기반으로 분석한다.

**Step 1 — 문서 읽기**
```
사용자가 제공한 파일 경로의 문서를 Read 도구로 읽는다.
여러 문서가 있으면 모두 읽는다.
```

**Step 2 — 도메인 요소 추출**

문서에서 아래 요소를 추출한다:
- **도메인 명사**: 반복 등장하는 비즈니스 개념 (잠재적 Aggregate/Entity)
- **도메인 동사**: 비즈니스 프로세스와 명령 (잠재적 Command/Event)
- **비즈니스 규칙**: "~이면", "~해야 한다", "~불가" 같은 조건/제약
- **이해관계자/조직**: 팀, 부서, 역할 (잠재적 컨텍스트 경계)
- **데이터 소유권**: 어느 시스템/팀이 어떤 데이터를 관리하는지

**Step 3 — 내부 추출 메모 작성** (0-A와 동일 형식)

### Phase 0 완료 체크포인트

추출 메모를 사용자에게 보여주고 확인한다:

## Phase 1: 서브도메인 식별 및 분류

`references/classification-criteria.md`의 분류 기준을 참고하여 각 후보 서브도메인을 분류한다.

### 분류 프로세스

1. **분류 표 작성**: 후보 서브도메인별로 아래 기준을 평가한다
   - 비즈니스 차별화 (상/중/하)
   - 복잡도 (상/중/하)
   - 변경 빈도 (높음/보통/낮음)
   - 외부 솔루션 대체 가능성 (가능/부분/불가)
   - **최종 분류**: Core / Supporting / Generic
   - **분류 근거** (1~2문장)

2. **Core Domain 수 검토**: Core가 3개 초과면 재검토를 권장한다
   > 핵심 도메인이 많으면 집중 투자 효과가 희석됩니다. 정말 비즈니스 경쟁의 원천인 것만 Core로 남기는 것이 좋습니다.

3. **사용자 확인**: 분류 표를 보여주고 수정 여부를 묻는다
   > "서브도메인 분류 초안입니다. 분류가 다르다고 생각하시는 항목이 있으면 알려주세요."

## Phase 2: 바운디드 컨텍스트 & 컨텍스트 맵

### 컨텍스트 도출

확정된 서브도메인을 기반으로 바운디드 컨텍스트를 도출한다. 기본 원칙:
- 유비쿼터스 언어가 달라지는 경계 = 컨텍스트 경계
- 팀 자율성이 필요한 단위 = 컨텍스트 경계
- 트랜잭션 독립성이 필요한 단위 = 컨텍스트 경계
- 하나의 서브도메인이 여러 컨텍스트로 나뉠 수도 있고, 여러 서브도메인이 하나의 컨텍스트일 수도 있다

### 관계 패턴 결정

각 컨텍스트 쌍에 대해 아래 패턴 중 하나를 적용한다:

| 패턴 | 선택 기준 |
|------|---------|
| **Customer-Supplier** | 상류가 하류의 요구를 반영할 의지가 있을 때 |
| **Conformist** | 하류가 상류 모델을 그대로 수용해도 될 때 |
| **ACL** | 레거시나 외부 시스템으로부터 도메인 모델을 보호해야 할 때 |
| **Open Host Service** | 하나의 컨텍스트가 여러 하류에 서비스를 제공할 때 |
| **Published Language** | 업계 표준 스키마나 프로토콜을 사용할 때 |
| **Shared Kernel** | 두 팀이 공유 코드를 함께 관리할 수 있을 때 |
| **Partnership** | 두 팀이 긴밀하게 함께 개발할 때 |
| **Separate Ways** | 통합 비용이 가치보다 클 때 |

### Mermaid 컨텍스트 맵 생성

컨텍스트와 관계를 Mermaid `graph TB` 형식으로 표현한다. Core/Supporting/Generic 그룹을 시각적으로 구분한다 (classDef 사용).

### 사용자 확인

컨텍스트 목록과 컨텍스트 맵 다이어그램을 보여주고 확인한다.

## Phase 3: 도메인 상세

### 3-1 유비쿼터스 언어 용어집

각 바운디드 컨텍스트별로 핵심 도메인 용어를 정리한다. 중요한 점:
- 같은 단어가 다른 컨텍스트에서 다른 의미로 쓰이면 반드시 표에서 비교해준다
- 코드 표현(클래스명/메서드명)도 함께 기록한다 (소스코드 모드에서 추출 가능)

### 3-2 도메인 이벤트 카탈로그

도메인 이벤트는 비즈니스 관점에서 의미 있는 상태 변화다:
- **이름 규칙**: 과거형 동사 + 명사 (예: `OrderPlaced`, `PaymentApproved`, `InventoryReserved`)
- 각 이벤트에 대해: 발행 컨텍스트, 트리거, 구독 컨텍스트, 핵심 페이로드 필드를 기술한다
- 주요 비즈니스 시나리오의 이벤트 연쇄를 Mermaid `sequenceDiagram`으로 시각화한다

### 3-3 애그리게이트 식별

각 바운디드 컨텍스트의 핵심 애그리게이트를 식별한다:
- **Root Entity**: 외부에서 참조하는 진입점 (ID를 가짐)
- **불변식**: 이 애그리게이트 안에서 항상 지켜야 하는 비즈니스 규칙
- **외부 참조**: 다른 애그리게이트는 ID만 참조 (객체 직접 참조 금지)
- **크기 경고**: 애그리게이트에 Entity가 5개 이상이면 분리를 검토한다

### 사용자 확인

용어집, 이벤트 카탈로그, 애그리게이트 목록을 보여주고 확인한다.

## Phase 4: 문서 생성 및 저장

### 문서 컴파일

`references/ddd-classification-template.md`의 템플릿을 기반으로 Phase 1~3에서 확정된 모든 내용을 채워 완성 문서를 생성한다.

**채워야 할 내용:**
- 메타데이터 (프로젝트명, 작성일, 입력 모드)
- 경영진 요약 (서브도메인 수, 주요 발견사항, 전략적 권고)
- 비즈니스 도메인 개요
- 서브도메인 분류표 + 상세 + Mermaid 사분면 차트
- 바운디드 컨텍스트 목록 + 상세
- 컨텍스트 맵 (Mermaid 다이어그램 + 관계 카탈로그)
- 유비쿼터스 언어 용어집 (컨텍스트별)
- 도메인 이벤트 카탈로그 + 이벤트 흐름 다이어그램
- 애그리게이트 식별 표
- 안티패턴 분석 (발견된 것만, 없으면 "발견되지 않음"으로 기재)
- 부록: 분석 소스 목록, 미해결 질문

마이그레이션 로드맵(섹션 10)은 기존 시스템 분석 시에만 포함하고, 신규 프로젝트 설계 시에는 생략한다.

### 품질 체크리스트

문서 생성 전 아래를 확인한다:

- [ ] 모든 서브도메인에 분류 근거가 명시되었는가
- [ ] Core Domain이 1~3개 범위인가 (많으면 재검토)
- [ ] 모든 바운디드 컨텍스트에 유비쿼터스 언어 용어가 최소 3개 이상인가
- [ ] 컨텍스트 맵에 모든 관계가 표현되었는가
- [ ] Mermaid 다이어그램 문법이 올바른가 (```mermaid 블록 사용)
- [ ] 도메인 이벤트가 과거형 명사로 명명되었는가
- [ ] 안티패턴 분석이 수행되었는가
- [ ] 부록에 분석 소스 목록이 있는가

### 파일 저장

```
저장 경로: docs/DDD-Classification-{프로젝트명}.md
(사용자가 다른 경로를 지정하면 그 경로 사용)
```

저장 후 사용자에게 알린다:
> "문서가 저장되었습니다: `docs/DDD-Classification-{프로젝트명}.md`
> 
> 이 문서를 바탕으로 특정 바운디드 컨텍스트의 전술적 설계(Aggregate 상세 설계, 헥사고날 아키텍처 패키지 구조 등)가 필요하시면 `/ddd` 스킬을 사용하세요."

## 주요 원칙

**입력 무관 동일 출력**
소스코드든 기존 문서든 동일한 템플릿 포맷의 문서를 생성한다. Phase 0이 정규화 레이어 역할을 하므로 Phase 1 이후 프로세스는 동일하다.

**질문 우선**
도메인 맥락이 불분명할 때는 추측하지 말고 사용자에게 묻는다. 특히 Core Domain 분류는 비즈니스 전략이 핵심이므로 반드시 확인한다.

**추적 가능성**
모든 분류 결정에는 근거를 기술한다. "왜 이렇게 나눴는가"를 나중에 읽는 사람이 이해할 수 있어야 한다.

**점진적 확정**
각 Phase가 완료되면 사용자 확인을 받는다. 한 번에 모두 완성하려 하지 말고, 단계별로 피드백을 반영한다.
