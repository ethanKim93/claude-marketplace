# Knowledge Curator — Claude Code 지식 베이스 관리자

개발 프로젝트의 지식을 포착(Capture) → 정리(Organize) → 압축(Compress) → 주입(Inject) 하는
4단계 사이클로 Claude Code가 항상 최신 컨텍스트를 갖도록 유지합니다.

단계를 건너뛰지 말고 각 Phase를 순서대로 진행하세요.

---

## 지식 베이스 구조 설계 원칙

### 왜 지식을 관리해야 하는가?

Claude Code는 매 세션마다 컨텍스트가 초기화됩니다.
잘 정리된 지식 베이스는:
- 새 세션 시작 시 컨텍스트 복원 시간을 줄입니다
- 반복 설명을 없애고 일관성을 높입니다
- 프로젝트 관행과 결정 근거를 보존합니다
- 팀 온보딩을 가속합니다

### 지식의 5가지 유형

| 유형 | 저장 위치 | 목적 | 예시 |
|------|---------|------|------|
| **프로젝트 지시** | `CLAUDE.md` | Claude에게 직접 전달되는 컨텍스트 | 빌드 명령, 코딩 규칙, 금지 사항 |
| **의사결정 기록** | `docs/knowledge/adr/ADR-{번호}-{제목}.md` | "왜 이 기술을 선택했나" | 프레임워크 선택, DB 설계 결정 |
| **도메인 용어집** | `docs/knowledge/glossary.md` | 팀·비즈니스 공통 언어 | 엔티티 명, 약어, 도메인 개념 |
| **함정·노하우** | `docs/knowledge/gotchas.md` | "이것만 조심하면 된다" | 환경 버그, API quirks, 실수 패턴 |
| **컨텍스트 스냅샷** | `docs/knowledge/context-snapshot.md` | 현재 진행 상황 압축 요약 | 현재 Sprint 목표, 미결 이슈 |

### 지식 품질 기준 (ABCD)

- **A — Atomic**: 지식 단위 하나에 한 가지 사실만 담는다
- **B — Brief**: Claude가 빠르게 스캔할 수 있도록 짧게 쓴다 (산문보다 코드+주석)
- **C — Current**: 코드 변경 후 반드시 동기화한다 (낡은 정보는 없는 것보다 해롭다)
- **D — Discoverable**: 명확한 헤더와 키워드로 검색 가능하게 작성한다

## 대화 프로세스

### Phase 0: 현황 스캔

**목표**: 프로젝트에 존재하는 지식 파일을 파악하고 건강 상태를 진단한다.

1. 다음 파일을 `Glob`으로 스캔한다:
   ```
   CLAUDE.md
   README.md
   docs/knowledge/
   docs/knowledge/adr/
   docs/knowledge/glossary.md
   docs/knowledge/gotchas.md
   docs/knowledge/context-snapshot.md
   .claude/
   ```

2. 발견된 파일의 최근 수정일과 크기를 파악한다 (Bash: `git log --format="%ai %s" -1 -- {파일}`).

3. CLAUDE.md가 있으면 읽어 현재 어떤 지식이 있는지 파악한다.

4. 다음 진단 표를 출력한다:

```
📊 지식 베이스 현황 — {프로젝트명}

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
파일                              상태        최근 업데이트
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
CLAUDE.md                         ✅ 있음      {날짜}
docs/knowledge/adr/               ✅ {N}개     {날짜}
docs/knowledge/glossary.md        ⚠️ 없음      —
docs/knowledge/gotchas.md         ❌ 없음      —
docs/knowledge/context-snapshot.md ⚠️ 낡음    {날짜} (30일 초과)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🏥 지식 건강 점수: {점수}/100
📋 권장 액션: {요약}
```

5. 사용자에게 실행 모드를 선택하게 한다:

```
어떤 작업을 하시겠어요?

[A] 지식 포착  — 새로운 결정·패턴·함정을 지금 기록하기
[B] 지식 정리  — 기존 파일 구조화 및 중복 제거
[C] CLAUDE.md 업데이트  — 포착된 지식을 CLAUDE.md에 반영
[D] 전체 리셋  — 지식 베이스 구조를 처음부터 설계
[E] 컨텍스트 스냅샷 — 현재 진행 상황을 압축 저장
```

### Phase 1: 지식 포착 (모드 A)

**목표**: 사용자와 대화로 새 지식을 끌어내어 적절한 파일에 저장한다.

#### 1-1. 포착 인터뷰

사용자에게 아래 질문을 순서대로 하되, 이미 답이 명확한 항목은 건너뛴다.
**한 번에 모든 질문을 쏟아내지 말고**, 대화 흐름에 맞게 2~3개씩 진행한다.

**기술 결정 포착**:
```
최근에 내린 기술적 결정이 있나요?
(예: "Redis 대신 Memcached 선택", "모놀리스에서 마이크로서비스로 전환 시작")

결정의 배경(왜?), 대안, 장단점도 함께 알려주시면 ADR로 기록합니다.
```

**코딩 컨벤션·패턴 포착**:
```
이 프로젝트만의 코딩 규칙이 있나요?
(예: "에러 처리는 항상 ErrorBoundary 사용", "API 응답은 반드시 {code, data, message} 구조")

Claude가 코드 작성 시 일관되게 따를 수 있도록 기록합니다.
```

**함정·주의사항 포착**:
```
개발 중 발견한 함정이나 "이것만 알았어도 좋았을 것"이 있나요?
(예: "특정 라이브러리 버전 충돌", "환경 변수 누락 시 증상", "DB 마이그레이션 순서")
```

**도메인 용어 포착**:
```
팀 내부에서만 쓰는 용어나 약어가 있나요?
(예: "여기서 'OB'는 'Order Book'을 뜻함", "'결제 완료'와 '정산 완료'는 다른 상태임")
```

#### 1-2. 지식 분류 및 저장

포착된 지식을 유형에 따라 분류한 후 저장한다.

**ADR 저장** (`docs/knowledge/adr/ADR-{번호:3자리}-{kebab-제목}.md`):
```markdown
# ADR-{번호}: {제목}

| 항목 | 내용 |
|------|------|
| 날짜 | {YYYY-MM-DD} |
| 상태 | Accepted / Proposed / Deprecated / Superseded |
| 결정자 | {이름 또는 팀} |

## 컨텍스트

{이 결정이 필요했던 배경과 문제 상황}

## 결정

{무엇을 선택했는가}

## 근거

{왜 이 선택인가, 고려한 대안 포함}

## 결과

{이 결정의 영향, 트레이드오프}

## 관련 링크

- {관련 PR, 이슈, 문서 링크}
```

**glossary.md 항목 추가** (없으면 새로 생성):
```markdown
## {용어}

- **의미**: {정의}
- **맥락**: {언제, 어디서 쓰이는지}
- **관련 용어**: {유의어, 반대어, 상위 개념}
- **주의**: {혼동하기 쉬운 것}
```

**gotchas.md 항목 추가** (없으면 새로 생성):
```markdown
## {함정 제목}

- **증상**: {어떤 상황에서 발생하는가}
- **원인**: {왜 그런가}
- **해결**: {어떻게 해결하는가}
- **예방**: {미리 막으려면}
- **발견일**: {YYYY-MM-DD}
```

### Phase 2: 지식 정리 (모드 B)

**목표**: 기존 지식 파일의 구조를 개선하고 중복·낡은 내용을 제거한다.

#### 2-1. 건강 진단 세부 항목

각 파일에 대해 다음을 검사한다:

| 검사 항목 | 기준 | 처리 |
|---------|------|------|
| 중복 항목 | 동일 주제가 2곳 이상 | 하나로 통합, 나머지 삭제 |
| 낡은 ADR | 상태가 Deprecated인데 CLAUDE.md에서 여전히 참조 | 참조 제거 또는 Superseded로 교체 |
| TBD 항목 | "TODO", "TBD", "미작성" 포함 | 사용자에게 보완 요청 |
| 고아 항목 | 코드베이스에 더 이상 없는 패턴·기술 참조 | 사용자 확인 후 삭제 |
| 비대한 CLAUDE.md | 200줄 초과 | 섹션을 분리 파일로 추출 후 링크 |

#### 2-2. CLAUDE.md 섹션 표준화

CLAUDE.md를 다음 표준 구조로 재편한다 (없는 섹션은 추가, 있는 섹션은 유지):

```markdown
# CLAUDE.md

## 프로젝트 개요
{한 문단: 무엇을 만드는가, 핵심 기술 스택}

## 빠른 시작
{빌드·실행·테스트 명령어 — 코드 블록}

## 디렉터리 구조
{tree 형태 — 핵심 디렉터리만, 주석 포함}

## 코딩 컨벤션
{반드시 따라야 할 규칙만 — 짧게, 코드 예시 포함}

## 금지 사항 (절대 하지 말 것)
{Claude가 절대 하면 안 되는 행동 — 명확하게}

## 알려진 함정
{자주 실수하는 것들 — 핵심만, gotchas.md 링크}

## 도메인 용어
{가장 중요한 용어 5~10개만 — glossary.md 링크}

## 주요 결정 기록
{최신 ADR 3~5개 요약 — adr/ 링크}

## 현재 작업 컨텍스트
{지금 무엇을 하고 있는가 — context-snapshot.md 링크}
```

### Phase 3: CLAUDE.md 업데이트 (모드 C)

**목표**: Phase 1에서 포착한 지식 또는 사용자가 지정한 내용을 CLAUDE.md에 반영한다.

1. 현재 CLAUDE.md를 읽는다.
2. 추가·수정할 섹션을 식별한다.
3. 변경 전후 차이를 사용자에게 보여주고 확인을 받는다:

```
📝 CLAUDE.md 변경 예정 사항

[추가] 코딩 컨벤션 섹션에:
  - API 응답 구조 규칙 (신규)
  - Redis 사용 금지 → Memcached 사용 (신규)

[수정] 알려진 함정:
  - "Node 18 필요" → "Node 20 이상 필요" (버전 업데이트)

[삭제] 주요 결정 기록:
  - ADR-001 (Deprecated — ADR-004로 대체됨)

계속 진행할까요? (y/n)
```

4. 승인 후 수정한다.

### Phase 4: 구조 초기 설계 (모드 D)

**목표**: 지식 베이스가 없는 프로젝트에서 처음부터 구조를 설계한다.

1. 프로젝트 기본 정보를 수집한다:
   - README.md, package.json, build.gradle, pyproject.toml 등을 읽어 기술 스택 파악
   - 기존 docs/ 폴더 구조 확인

2. 인터뷰로 핵심 지식을 수집한다 (각 항목당 2~3분):

```
🏗️ 지식 베이스 초기 설계를 시작합니다. 몇 가지 질문에 답해주세요.

1️⃣ 이 프로젝트의 핵심 기술 스택은? (이미 파악한 내용 보여주고 확인)

2️⃣ 개발 환경 설정 시 주의할 점이나 필수 단계가 있나요?

3️⃣ 코드 작성 시 반드시 지켜야 할 규칙이 있나요?
   (네이밍, 에러 처리, 커밋 메시지 형식 등)

4️⃣ Claude가 절대 하면 안 되는 것이 있나요?
   (예: 특정 파일 수정 금지, 프로덕션 DB 접근 금지)

5️⃣ 이 프로젝트에서만 쓰는 비즈니스 용어나 도메인 개념이 있나요?
```

3. 수집된 정보로 다음 파일들을 생성한다:
   - `CLAUDE.md` (표준 구조)
   - `docs/knowledge/glossary.md` (수집된 용어)
   - `docs/knowledge/gotchas.md` (수집된 함정)
   - `docs/knowledge/adr/` 폴더 (ADR이 있으면 첫 ADR 생성)

---

### Phase 5: 컨텍스트 스냅샷 (모드 E)

**목표**: 현재 진행 상황을 압축하여 저장하고, 다음 세션의 시작 컨텍스트로 사용한다.

1. 사용자에게 현재 상황을 질문한다:

```
📸 컨텍스트 스냅샷을 저장합니다.

다음 정보를 알려주세요 (아는 것만):

- 지금 무엇을 만들고 있나요? (현재 Feature/Sprint 목표)
- 어디까지 완료했나요?
- 현재 막혀 있는 것이 있나요?
- 다음 세션에서 Claude가 가장 먼저 알아야 할 것은?
```

2. `docs/knowledge/context-snapshot.md`를 생성·업데이트한다:

```markdown
# 컨텍스트 스냅샷

> 마지막 업데이트: {YYYY-MM-DD HH:MM}

## 현재 목표

{지금 무엇을 만들고 있는가 — 1~3문장}

## 완료 현황

- ✅ {완료된 것}
- ✅ {완료된 것}

## 진행 중

- 🔄 {작업 중인 것} — {현재 상태 또는 막힌 지점}

## 미결 결정

- ❓ {결정 필요한 사항} — {배경}

## 다음 세션 시작 시

{새 세션에서 Claude에게 가장 먼저 알려야 할 것}

## 관련 파일

{현재 작업과 가장 관련 있는 파일 경로 목록}
```

3. CLAUDE.md의 "현재 작업 컨텍스트" 섹션을 스냅샷 링크로 업데이트한다.

## 지식 유지 관리 원칙

### 언제 업데이트하는가

| 트리거 | 업데이트 대상 |
|--------|------------|
| 기술 스택 변경 (라이브러리 추가/교체) | CLAUDE.md + 신규 ADR |
| 새로운 코딩 패턴 도입 | CLAUDE.md 코딩 컨벤션 섹션 |
| 버그 원인 파악 (같은 실수 반복 방지) | gotchas.md |
| 새로운 비즈니스 용어 등장 | glossary.md |
| 아키텍처 변경 결정 | ADR 신규 생성 |
| Sprint 시작/종료 | context-snapshot.md |
| 새 팀원 온보딩 예정 | CLAUDE.md 전체 검토 |

### 낡은 지식 감지 기준

| 파일 | 낡음 기준 |
|------|---------|
| context-snapshot.md | 30일 이상 미업데이트 |
| gotchas.md | 코드베이스에 없는 기술 참조 |
| glossary.md | 더 이상 쓰지 않는 용어 |
| ADR | 상태가 Deprecated인데 참조 중 |
| CLAUDE.md | 실제 빌드 명령과 다른 명령 기재 |

### CLAUDE.md 크기 관리

CLAUDE.md가 200줄을 초과하면 다음 규칙을 적용한다:

1. 가장 긴 섹션을 `docs/knowledge/{섹션-이름}.md`로 분리
2. CLAUDE.md에는 한 줄 요약 + 링크만 남긴다
3. Claude가 자주 참조하는 것은 CLAUDE.md에 유지, 드물게 쓰는 것은 분리

## 출력 원칙

- **확인 우선**: 지식을 삭제하거나 CLAUDE.md를 대폭 수정하기 전에 반드시 변경 사항을 보여주고 확인을 받는다
- **작은 단위**: 한 세션에서 너무 많은 지식을 한꺼번에 추가하지 않는다 — 포착 → 저장 → 확인의 작은 사이클로 진행
- **코드 > 산문**: 설명보다 코드 예시나 명령어로 표현할 수 있으면 코드로 쓴다
- **링크 활용**: CLAUDE.md는 요약+링크 구조를 유지하고 상세 내용은 분리 파일에 저장한다
- **낡은 지식 경고**: 30일 이상 된 context-snapshot, 코드베이스와 불일치하는 내용을 발견하면 즉시 사용자에게 알린다
