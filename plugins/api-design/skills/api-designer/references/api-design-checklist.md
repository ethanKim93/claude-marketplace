# API 설계 체크리스트 & 안티패턴 진단

## 24개 설계 체크포인트

### URL & 리소스 설계 (7개)

- [ ] **URL-1** 모든 리소스명이 복수 명사인가? (`/users` not `/user`)
- [ ] **URL-2** URL이 소문자 + 하이픈(kebab-case)인가? (`/user-profiles`)
- [ ] **URL-3** URL에 동사가 없는가? (`/createUser` → `/users`)
- [ ] **URL-4** 버전이 URL에 포함되어 있는가? (`/api/v1/...`)
- [ ] **URL-5** 계층 깊이가 3단계 이하인가? (4단계 이상 재검토)
- [ ] **URL-6** 상태 전환이 서브 리소스로 모델링되어 있는가? (`/activations`)
- [ ] **URL-7** Path Parameter 명이 일관성 있는가? (`{userId}`, `{orderId}` 등)

### HTTP 메서드 & 상태 코드 (5개)

- [ ] **HTTP-1** 메서드 사용이 시맨틱에 맞는가? (GET=조회, POST=생성, PUT=교체, PATCH=수정, DELETE=삭제)
- [ ] **HTTP-2** 생성 성공이 `201 Created` + `Location` 헤더를 반환하는가?
- [ ] **HTTP-3** 삭제 성공이 `204 No Content`를 반환하는가?
- [ ] **HTTP-4** 인증 실패(401)와 권한 없음(403)이 구분되어 있는가?
- [ ] **HTTP-5** 비즈니스 유효성 실패가 `422`를 반환하는가? (400은 형식 오류)

### 요청/응답 설계 (5개)

- [ ] **REQ-1** 페이지네이션이 구현되어 있는가? (목록 API)
- [ ] **REQ-2** 에러 응답이 RFC 7807 Problem Details 형식인가?
- [ ] **REQ-3** 날짜/시간이 ISO 8601 UTC 형식인가? (`2024-01-15T09:00:00Z`)
- [ ] **REQ-4** 응답에 민감 정보가 포함되어 있지 않은가? (password, secretKey 등)
- [ ] **REQ-5** 멱등성이 필요한 POST에 Idempotency-Key 지원이 있는가?

### 보안 (4개)

- [ ] **SEC-1** 모든 엔드포인트에 인증이 적용되어 있는가? (퍼블릭 제외)
- [ ] **SEC-2** 인가(Authorization) 로직이 있는가? (내 데이터만 접근 가능한지)
- [ ] **SEC-3** Rate Limiting이 구현되어 있는가? (`X-RateLimit-*` 헤더 포함)
- [ ] **SEC-4** 500 에러에 내부 정보(스택 트레이스, DB 오류)가 노출되지 않는가?

### OpenAPI 문서화 (3개)

- [ ] **DOC-1** 모든 엔드포인트에 `summary`와 `operationId`가 있는가?
- [ ] **DOC-2** 공통 컴포넌트(`$ref`)가 활용되어 중복이 최소화됐는가?
- [ ] **DOC-3** 모든 에러 응답(4xx, 5xx)이 문서화되어 있는가?

## 26개 안티패턴 진단표

### 카테고리 1: URL 안티패턴

| # | 안티패턴 | 증상 | 수정 방법 |
|---|----------|------|-----------|
| 1 | **동사 URL** | `POST /createUser`, `GET /getOrders` | `POST /users`, `GET /orders` |
| 2 | **단수 컬렉션** | `GET /user`, `POST /product` | `GET /users`, `POST /products` |
| 3 | **버전 누락** | `/api/users` (v 없음) | `/api/v1/users` |
| 4 | **대문자/언더스코어** | `/UserProfiles`, `/user_profiles` | `/user-profiles` |
| 5 | **파일 확장자** | `/users.json`, `/report.csv` | `/users` + `Accept` 헤더 |
| 6 | **과도한 중첩** | `/a/{id}/b/{id}/c/{id}/d/{id}` (4단계+) | 독립 리소스로 분리 |
| 7 | **동작 서브 URL** | `/users/{id}/get`, `/orders/{id}/fetch` | `GET /users/{id}` |

### 카테고리 2: HTTP 메서드 안티패턴

| # | 안티패턴 | 증상 | 수정 방법 |
|---|----------|------|-----------|
| 8 | **모든 것이 POST** | 조회도 `POST /getUser` | `GET /users/{id}` |
| 9 | **모든 것이 GET** | 생성도 `GET /createUser?name=foo` | `POST /users` |
| 10 | **PUT으로 부분 수정** | `PUT /users/{id}` + 일부 필드만 전송 | `PATCH /users/{id}` |
| 11 | **DELETE에 바디** | 삭제 필터를 바디로 전달 | `DELETE /resources?ids=1,2,3` 또는 배치 API |

### 카테고리 3: 상태 코드 안티패턴

| # | 안티패턴 | 증상 | 수정 방법 |
|---|----------|------|-----------|
| 12 | **에러도 200** | 실패 시에도 `200 OK` + 에러 바디 | 적절한 4xx/5xx 사용 |
| 13 | **생성도 200** | `POST`로 생성 후 `200 OK` | `201 Created` + `Location` |
| 14 | **삭제도 200** | `DELETE` 성공 후 바디와 함께 `200` | `204 No Content` |
| 15 | **400 남용** | 모든 에러가 `400 Bad Request` | 401, 403, 404, 409, 422 구분 |
| 16 | **비표준 상태 코드** | `418 I'm a teapot` 생산 사용, `700` 등 | RFC 9110 표준 코드만 |

### 카테고리 4: 응답 설계 안티패턴

| # | 안티패턴 | 증상 | 수정 방법 |
|---|----------|------|-----------|
| 17 | **페이지네이션 없는 목록** | 목록 API가 모든 데이터 반환 | `page`/`limit` 또는 `cursor` 추가 |
| 18 | **비일관 응답 구조** | 어떤 API는 envelope, 어떤 API는 plain | 전체 통일 |
| 19 | **민감 정보 노출** | 응답에 `password`, `secretKey`, `cvv` | 응답 DTO에서 민감 필드 제거 |
| 20 | **N+1 자동 포함** | 항상 연관 객체 전체 반환 | `?include=` 파라미터로 선택적 포함 |
| 21 | **비표준 날짜 형식** | `"2024/01/15"`, Unix timestamp | ISO 8601: `"2024-01-15T09:00:00Z"` |
| 22 | **마법 숫자** | `"status": 1` (1이 뭔지 모름) | `"status": "ACTIVE"` 열거형 문자열 |

### 카테고리 5: 보안 안티패턴

| # | 안티패턴 | 증상 | 수정 방법 |
|---|----------|------|-----------|
| 23 | **순차 ID 노출** | `id: 1, 2, 3` (비즈니스 정보 추론 가능) | UUID, KSUID, Snowflake ID 사용 |
| 24 | **인증 없는 엔드포인트** | API Key/토큰 없이도 데이터 접근 가능 | 모든 엔드포인트에 인증 미들웨어 |
| 25 | **API Key 쿼리 파라미터** | `/api/users?api_key=secret123` (로그 노출) | `Authorization` 또는 `X-API-Key` 헤더 |
| 26 | **500에 내부 정보 노출** | 스택 트레이스, DB 쿼리, 서버 경로 노출 | `traceId`만 반환, 서버 로그에서 추적 |

## 설계 품질 점수 계산

```
체크리스트 통과 수 / 24 × 100 = 기본 점수

안티패턴 감점:
- 카테고리 1~2 (URL/메서드): 각 -3점
- 카테고리 3 (상태 코드): 각 -3점
- 카테고리 4 (응답 설계): 각 -2점
- 카테고리 5 (보안): 각 -5점

점수 해석:
90~100: 우수 — 프로덕션 준비 완료
75~89:  양호 — 소수 개선 필요
60~74:  보통 — 주요 이슈 수정 필요
~59:    미흡 — 설계 재검토 필요
```

## 빠른 진단 쿼리

기존 OpenAPI 파일을 리뷰할 때:

```bash
# 동사 URL 패턴 찾기
grep -E "/(get|create|update|delete|fetch|retrieve|add|remove|set)[A-Z/]" api.yaml

# 버전 없는 경로 찾기
grep -E "^  /[a-z]" api.yaml | grep -v "/api/v"

# 단수 리소스 패턴 (간이 검사)
grep -E "/[a-z]+/{" api.yaml | grep -v "s/{" | grep -v "ss/{"
```

## Richardson Maturity Model 자가 평가

| 레벨 | 조건 | 충족 여부 |
|------|------|-----------|
| **Level 0** | HTTP 사용 | - |
| **Level 1** | 리소스별 고유 URL | URL 안티패턴 0개 |
| **Level 2** | HTTP 메서드 + 상태 코드 올바른 사용 | 메서드·상태코드 안티패턴 0개 |
| **Level 3** | HATEOAS (응답에 링크 포함) | 선택 사항 — 필요 시만 구현 |

**현실적 목표**: Level 2 완전 충족 + 보안 항목 100% 통과
