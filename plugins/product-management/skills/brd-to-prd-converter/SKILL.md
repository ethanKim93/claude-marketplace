# BRD → PRD 변환기 스킬

승인된 BRD를 분석하고, PRD 초안을 자동으로 생성합니다.
단계를 건너뛰지 말고 각 Phase를 순서대로 진행하세요.

---

## 변환 철학

BRD와 PRD는 같은 프로젝트를 서로 다른 독자에게 설명한다.

- **BRD** → "왜(Why)" + "무엇(What, 비즈니스 관점)" — 경영진·스폰서·사업부서
- **PRD** → "무엇(What, 제품 관점)" + "어떻게 검증하는가" — PM·디자이너·개발팀

변환의 핵심은 **BRD의 비즈니스 목표를 사용자 스토리와 기능 요구사항으로 번역**하는 것이다.

### BRD → PRD 매핑 규칙

| BRD 섹션 | PRD 섹션 | 변환 방식 |
|----------|----------|-----------|
| 섹션 3: 프로젝트 목표 (SMART) | 목표 & KPI | 목표 그대로 유지, KPI에 측정 기준값 추가 요청 |
| 섹션 2: 비즈니스 배경 | 문제 정의 | 비즈니스 문제 → 사용자 관점 문제로 재서술 |
| 섹션 4: 범위 | 범위 & Non-goal | In-scope/Out-of-scope 직접 매핑 |
| 섹션 5: 이해관계자 | 타겟 사용자 | 최종 사용자 → 페르소나로 전환 |
| 섹션 6: 비즈니스 요구사항 (REQ-NNN) | 기능 요구사항 | REQ-ID를 FR-ID로 재매핑, P0/P1/P2 우선순위 부여 |
| 섹션 8: 리스크 | 리스크 | 그대로 유지, 기술적 리스크 추가 요청 |
| 섹션 9: 일정 & 마일스톤 | 타임라인 | 마일스톤 그대로, 담당자 추가 요청 |
| 섹션 7: 제약 조건 & 가정 | 의존성 & 가정 | 제약 → 비기능 요구사항 후보로 분류 |

## 대화 프로세스

### Phase 0: BRD 문서 로드 및 분석

**목표**: BRD 파일을 읽고 핵심 정보를 추출한다.

1. 사용자가 BRD 파일 경로를 제공했는지 확인한다.
   - 경로 제공 → `Read`로 파일을 읽는다.
   - 경로 없음 → `Glob`으로 `docs/*/BRD.md` 패턴을 스캔하여 기존 도메인을 탐색한다.
     - 도메인이 하나뿐이면 자동 선택하고 사용자에게 알린다.
     - 도메인이 여러 개이면 목록을 보여주고 사용자에게 선택을 요청한다.
   - 파일 없음 → 사용자에게 경로 또는 내용 붙여넣기를 요청한다.

2. BRD에서 다음 항목을 추출하고 내부 메모로 정리한다:

   ```
   [추출 항목]
   - 프로젝트명
   - 비즈니스 배경 (섹션 2)
   - SMART 목표 목록 (섹션 3)
   - In-scope / Out-of-scope (섹션 4)
   - 이해관계자 목록 → 최종 사용자 역할만 필터링 (섹션 5)
   - REQ-ID 요구사항 목록 + MoSCoW 등급 (섹션 6)
   - 제약 조건 & 가정 (섹션 7)
   - 리스크 목록 (섹션 8)
   - 마일스톤 (섹션 9)
   ```

3. BRD 품질 사전 체크:
   - REQ-ID 체계 없음 → 경고 후 임시 ID 자동 부여
   - SMART 목표 없음 → Phase 1에서 사용자에게 보완 요청
   - Must Have > 60% → 경고 메시지 출력

4. 추출 결과를 사용자에게 보고 후 PRD 포맷 선택을 물어본다.

---

### Phase 1: PRD 포맷 선택

**목표**: 프로젝트 성격에 맞는 PRD 포맷을 결정한다.

사용자에게 다음 세 가지 중 선택을 요청한다:

| 포맷 | 적합한 상황 | 섹션 수 |
|------|------------|---------|
| **표준 PRD** | 팀 규모 5명 이상, 개발 기간 3개월 이상 | 13섹션 |
| **경량 PRD** | 소규모 팀, 빠른 피처 개발, 2주~1개월 | 5섹션 |
| **PR-FAQ** | 신규 제품/서비스, 아마존 스타일, 사용자 관점 강조 | 보도자료 + FAQ |

BRD 규모(섹션 수, 요구사항 수)를 바탕으로 추천 포맷을 제안한다.

---

### Phase 2: 섹션별 변환 실행

**목표**: BRD 정보를 PRD 각 섹션으로 자동 변환한다.

#### 2-1. 문제 정의 변환

BRD 섹션 2(비즈니스 배경)를 사용자 관점으로 재서술:

```
변환 전 (BRD): "현재 수동 승인 프로세스로 인해 처리 시간이 평균 5일 소요되어 운영 비용이 연 2억 원 발생"
변환 후 (PRD): "담당자는 승인 요청 후 결과를 5일 이상 기다려야 하며, 처리 상태를 실시간으로 확인할 수 없어 반복 문의가 발생함"
```

#### 2-2. 목표 & KPI 변환

BRD SMART 목표 → PRD KPI 테이블:

| BRD 목표 | PRD KPI | 현재값 | 목표값 | 측정 방법 |
|---------|---------|-------|-------|---------|
| (SMART 목표 그대로) | (측정 지표명) | _TBD_ | (목표 수치) | (측정 도구) |

현재값(Baseline)이 BRD에 없으면 `_TBD_`로 표시하고 사용자에게 확인 요청.

#### 2-3. 타겟 사용자 → 페르소나 변환

BRD 이해관계자 중 최종 사용자만 필터링하여 2-3개 페르소나 초안 생성:

```markdown
### 페르소나 1: {역할명}
- **배경**: {BRD 이해관계자 설명 기반}
- **주요 목표**: {BRD 범위에서 추론}
- **페인포인트**: {BRD 비즈니스 배경에서 추론}
- **보완 필요**: 나이·기술 친숙도·사용 빈도 등 (사용자 인터뷰 필요)
```

#### 2-4. REQ-ID → 기능 요구사항 변환

BRD REQ-NNN을 기능(Feature)별로 그룹핑한 뒤 PRD FR-NNN으로 재매핑한다.

1. BRD REQ 목록을 도메인 기능 단위로 묶는다. 기능 경계가 불명확하면 사용자에게 확인한다.
2. 각 기능 그룹을 섹션 7의 `### 7.N Feature: {기능명-kebab-case}` 챕터로 구성한다.
   ```
   ### 7.1 Feature: {기능명-kebab-case}
   | FR-ID | 요구사항 | 우선순위 |
   ```
3. 챕터별 FR 테이블을 작성한다:

| BRD REQ-ID | PRD FR-ID | 기능 설명 | 우선순위 | 인수 조건 초안 |
|-----------|-----------|---------|--------|--------------|
| REQ-001 (Must) | FR-001 | (요구사항 설명) | P0 | Given…When…Then… |
| REQ-002 (Should) | FR-002 | (요구사항 설명) | P1 | _작성 필요_ |
| REQ-003 (Could) | FR-003 | (요구사항 설명) | P2 | _작성 필요_ |

**MoSCoW → P레벨 매핑**: Must→P0, Should→P1, Could→P2, Won't→비포함(Non-goal)

P0 요구사항에 대해서는 Given-When-Then 인수 조건 초안을 자동 생성한다.
P1/P2는 `_작성 필요_`로 표시하여 prd-writer로 보완하도록 안내한다.

#### 2-5. 범위 직접 매핑

BRD In-scope → PRD In-scope (그대로)
BRD Out-of-scope + BRD Won't Have → PRD Non-goal (통합)

#### 2-6. 비기능 요구사항 초안

BRD 제약 조건(섹션 7)과 NFR(섹션 6)에서 추출:
- 성능·보안·가용성 수치가 있으면 그대로 이전
- 없으면 카테고리만 나열하고 `_수치 확인 필요_`로 표시

### Phase 3: 품질 검증 & 보완 질문 도출

**목표**: 변환 결과의 누락 항목을 파악하고 사용자가 채워야 할 부분을 명시한다.

자동 변환 후 다음 항목을 체크한다:

| 체크 항목 | 상태 | 액션 |
|---------|------|------|
| 페르소나에 기술 친숙도가 있는가 | ⚠️ | 사용자에게 확인 요청 |
| KPI에 현재 기준값(Baseline)이 있는가 | ⚠️ | 데이터 팀 확인 필요 |
| P0 요구사항 모두 인수 조건이 있는가 | ✅/❌ | 없으면 자동 생성 |
| 외부 시스템 의존성이 명시되었는가 | ✅/❌ | BRD 섹션 7에서 추출 |
| 사용자 스토리가 있는가 | ❌ | Phase 2에서 신규 작성 필요 |

> 사용자 스토리는 BRD에 없는 항목으로, 이 단계에서 FR 목록을 바탕으로 초안을 생성한다.

**보완 질문 출력 예시:**

```
🔍 변환 완료 — 다음 항목을 보완해야 합니다:

[High Priority]
Q1. 현재 처리 시간(Baseline)이 정확히 몇 분/초인지 측정된 데이터가 있나요?
    → KPI 현재값을 채우기 위해 필요합니다.

[Medium Priority]
Q2. 페르소나 1(담당자)의 평균 기술 친숙도는 어느 정도인가요? (1-5 척도)
    → 솔루션 복잡도 결정에 영향을 줍니다.

[Low Priority]
Q3. P1 요구사항 FR-004의 인수 조건을 작성해 주실 수 있나요?
    → 로드맵 생성 전에 있으면 더 정확한 일정 산출이 가능합니다.
```

### Phase 4: PRD 초안 저장

**목표**: 변환된 PRD를 파일로 저장하고 다음 단계를 안내한다.

1. `docs/{domain}/PRD.md`로 저장한다. (`{domain}`은 Phase 0에서 확정한 도메인명)

2. 파일 상단에 변환 메타데이터를 기록한다:

```markdown
> **변환 정보**
> - 원본 BRD: `docs/{domain}/BRD.md`
> - 변환일: {YYYY-MM-DD}
> - BRD REQ-ID → PRD FR-ID 매핑: {N}개 요구사항 변환됨
> - 보완 필요 항목: {N}개 (_TBD_ 표시)
```

3. 저장 완료 후 권장 다음 단계를 안내한다:

```
✅ PRD 초안이 생성되었습니다: docs/{domain}/PRD.md

📋 권장 다음 단계:
1. /prd-writer — 보완 질문의 답변을 반영하여 PRD 완성
2. /prd-reviewer — 완성된 PRD 품질 검토 (다관점 리뷰)
3. 팀 공유 후 /roadmap-generator로 로드맵 생성
```

## 변환 품질 원칙

- **추적성 유지**: 모든 FR-ID는 원본 BRD REQ-ID와 연결을 기록한다.
- **가정 명시**: BRD에 없어서 추론한 내용은 ⚠️ 또는 `_추론_` 표시로 명확히 구분한다.
- **_TBD_ 규칙**: 데이터 없이 채울 수 없는 칸은 반드시 `_TBD_`로 표시 — 빈 칸보다 낫다.
- **재실행 권장**: 보완 질문 답변 후 `/prd-writer`로 세부 내용을 채우고 `/prd-reviewer`로 검토한다.
