# DDD → BRD 변환기 스킬

DDD 도메인 분류 문서를 분석하고, **전체(Master) BRD 1개 + 도메인별 BRD**를 자동 생성합니다.
단계를 건너뛰지 말고 각 Phase를 순서대로 진행하세요.

---

## 변환 철학

DDD 분류 문서와 BRD는 같은 시스템을 서로 다른 독자에게 설명한다.

- **DDD 분류 문서** → "무엇이 존재하는가(도메인 구조)" — 아키텍트·개발팀
- **BRD** → "왜 만드는가(비즈니스 가치) + 무엇을 해야 하는가(요구사항)" — 경영진·사업팀

변환의 핵심은 **도메인 구조를 비즈니스 목표와 요구사항으로 번역**하는 것이다.

### DDD → BRD 매핑 규칙

| DDD 분류 문서 섹션 | BRD 섹션 | 변환 방식 |
|---|---|---|
| 섹션 1: 경영진 요약 (전략 권고) | 섹션 1: 경영진 요약 + 섹션 3: 프로젝트 목표 | 권고 → SMART 목표 초안 (수치는 사용자 확인 필요) |
| 섹션 2: 비즈니스 도메인 개요 | 섹션 2: 비즈니스 배경 & 필요성 | 거의 직접 매핑 |
| 섹션 3: 서브도메인 분류 (Core/Supporting/Generic) | 섹션 4: 프로젝트 범위 (In/Out-scope) | Core → In-scope, Generic → Out-scope/외부 솔루션 권고 |
| 섹션 5: 컨텍스트 맵 (통합 패턴) | 섹션 6.2: 비기능 요구사항 (통합) | 통합 패턴 → 연동 요구사항 초안 |
| 섹션 6: 유비쿼터스 언어 | 섹션 12: 용어 사전 | 거의 직접 매핑 (코드 표현 제거) |
| 섹션 7: 도메인 이벤트 카탈로그 | 섹션 6.1: 기능 요구사항 | 이벤트 → "시스템은 ~해야 한다" REQ 변환 |
| 섹션 8: 애그리게이트 불변식 | 섹션 6.1: 기능 요구사항 (비즈니스 규칙) | 불변식 → 제약 요구사항 |
| 섹션 9: 안티패턴 분석 | 섹션 8: 리스크 & 완화 방안 | 안티패턴 → 기술 리스크로 전환 |

## 출력 파일 구조

```
docs/
├── _project/
│   ├── BRD-master.md               # 전체 BRD (도메인별 링크 포함)
│   └── BRD-supplement.md           # 보완 설문서 (Phase 1에서 생성)
├── {domain-1}/
│   └── BRD.md                      # Core 도메인 BRD (도메인명은 DDD 서브도메인명을 kebab-case로 변환)
├── {domain-2}/
│   └── BRD.md                      # Core 도메인 BRD
└── {domain-3}/
    └── BRD.md                      # Supporting 도메인 BRD (선택)
```

**도메인 BRD 생성 기준:**
- **Core 도메인**: 항상 개별 BRD 생성
- **Supporting 도메인**: 사용자 선택 (기본: 생성)
- **Generic 도메인**: 개별 BRD 미생성, Master BRD에 "외부 솔루션 권고"로 기재

## 대화 프로세스

### Phase 0: DDD 문서 로드 및 분석

**목표**: DDD 분류 문서를 읽고 BRD 작성에 필요한 정보를 추출한다.

1. 사용자가 DDD 분류 문서 경로를 제공했는지 확인한다.
   - 경로 제공 → `Read`로 파일을 읽는다.
   - 경로 없음 → `Glob`으로 `docs/_project/DDD-Classification.md` 파일을 검색한다.
   - 파일 없음 → 사용자에게 `/ddd-classifier` 스킬을 먼저 실행하도록 안내한다.

2. DDD 문서에서 다음 항목을 추출하고 내부 메모로 정리한다:

   ```
   [추출 항목]
   - 프로젝트명
   - 비즈니스 도메인 개요 (섹션 2)
   - 전략적 권고사항 (섹션 1)
   - 서브도메인 목록 + Core/Supporting/Generic 분류 (섹션 3)
   - 바운디드 컨텍스트 목록 + 담당팀/소유권 힌트 (섹션 4)
   - 컨텍스트 맵 관계 (섹션 5)
   - 유비쿼터스 언어 용어집 (섹션 6)
   - 도메인 이벤트 카탈로그 (섹션 7)
   - 애그리게이트 불변식 (섹션 8)
   - 안티패턴 목록 (섹션 9)
   - 마이그레이션 로드맵 단계 (섹션 10, 있을 경우)
   ```

3. 추출 결과를 아래 형식으로 사용자에게 보고한다:

   ```
   ## DDD 문서 분석 결과

   **프로젝트명**: {명}
   **서브도메인**: {N}개 (Core: {N}, Supporting: {N}, Generic: {N})
   **바운디드 컨텍스트**: {N}개
   **도메인 이벤트**: {N}개
   **안티패턴**: {N}개 발견

   ### 생성 예정 BRD 파일
   - docs/_project/BRD-master.md (전체)
   - docs/{core-domain-1}/BRD.md
   - docs/{core-domain-2}/BRD.md
   (Supporting 도메인 포함 여부는 다음 단계에서 확인)

   ### 자동 채움 가능 항목 (약 50%)
   ✅ 비즈니스 배경 — DDD 섹션 2에서 추출
   ✅ 프로젝트 범위 — 서브도메인 분류에서 매핑
   ✅ 기능 요구사항 초안 — 도메인 이벤트 {N}개 변환
   ✅ 비기능 요구사항(통합) — 컨텍스트 맵에서 추출
   ✅ 리스크 초안 — 안티패턴 {N}개 변환
   ✅ 용어 사전 — 유비쿼터스 언어에서 추출

   ### 사용자 보완 필요 항목 (약 50%)
   ⚠️ 프로젝트 목표 (SMART 수치화)
   ⚠️ 이해관계자 분석
   ⚠️ 예산 & 비용-편익
   ⚠️ 일정 & 마일스톤
   ⚠️ 기능 요구사항 MoSCoW 우선순위
   ```

4. Supporting 도메인 BRD 포함 여부를 사용자에게 확인한다.

### Phase 2: BRD 콘텐츠 생성

**목표**: 추출한 DDD 정보와 사용자 보완 정보를 BRD 섹션으로 변환한다.

#### 2-1. 비즈니스 배경 변환

DDD 섹션 2(비즈니스 도메인 개요)를 BRD 섹션 2로 변환:
- 비즈니스 역량 테이블 → As-Is 상태 (현재 문제점)
- Core Domain 전략 방향 → To-Be 상태
- 비즈니스 드라이버: DDD 분류가 필요했던 배경을 "왜 지금인가"로 재서술

#### 2-2. 프로젝트 범위 매핑

서브도메인 분류를 BRD 섹션 4로 변환:

```
Core 도메인 → In-scope (핵심 개발 대상)
Supporting 도메인 → In-scope (자체 개발 또는 경량 솔루션)
Generic 도메인 → Out-scope + 비고: "상용 솔루션 또는 SaaS 활용 권고"
```

마이그레이션 로드맵이 있으면 단계별로 In-scope를 Phase 1/2/3으로 분리한다.

#### 2-3. 도메인 이벤트 → 기능 요구사항 변환

각 도메인 이벤트를 REQ-NNN 형식으로 변환한다:

| 변환 규칙 |
|---------|
| 이벤트 이름이 `OrderPlaced`이면 → "시스템은 주문을 등록할 수 있어야 한다" |
| 이벤트 구독 컨텍스트가 있으면 → "시스템은 {이벤트} 발생 시 {구독 컨텍스트}에 알림을 전송해야 한다" |
| 애그리게이트 불변식 → "시스템은 {불변식 조건}을 강제해야 한다" |

```markdown
#### [{바운디드 컨텍스트명}]

| 요구사항 ID | 설명 | 우선순위 | 출처 |
|----------|------|---------|-----|
| REQ-{NNN} | {이벤트 기반 요구사항} | _TBD_ | DE-{NN} |
| REQ-{NNN} | {불변식 기반 요구사항} | _TBD_ | AG-{NN} |
```

- 우선순위는 설문서 섹션 E를 채운 경우 반영하고, 미작성 시 `_TBD_`로 표시
- Core 도메인 요구사항은 Must Have 초기값으로 설정, 사용자 확인 권장

#### 2-4. 컨텍스트 맵 → 비기능 요구사항(통합) 변환

각 컨텍스트 관계(CM-NN)를 통합 요구사항으로 변환:

| 패턴 | 변환 NFR |
|-----|---------|
| Customer-Supplier, Open Host | 상류 컨텍스트 API 연동 요구사항 |
| ACL (Anti-Corruption Layer) | 외부 시스템 격리 요구사항 (데이터 변환 레이어 필요) |
| Published Language | 업계 표준 스키마 준수 요구사항 |

#### 2-5. 안티패턴 → 리스크 변환

각 안티패턴(AP-NN)을 RISK-NNN으로 변환:

```markdown
| 리스크 ID | 설명 | 발생 가능성 | 영향도 | 우선순위 | 완화 방안 | 담당자 |
|---------|------|-----------|------|--------|---------|------|
| RISK-{NNN} | {안티패턴 → 기술 리스크로 재서술} | {AP 우선순위 기반} | 높음 | 1 | {AP 권고 액션} | _TBD_ |
```

#### 2-6. 도메인별 BRD 범위 결정

각 도메인 BRD는 Master BRD의 부분집합이다:

```
도메인 BRD = Master BRD의 해당 도메인 섹션만 추출
  + 해당 Bounded Context 상세
  + 해당 도메인 요구사항 (REQ-NNN 중 해당 BC 출처만)
  + 해당 도메인 유비쿼터스 언어
  + 해당 도메인 이벤트 흐름 다이어그램 (DDD 섹션 7에서 추출)
  + Master BRD 역참조 링크
```

### Phase 3: 품질 체크

**목표**: 생성 전 콘텐츠 품질을 확인한다.

| 체크 항목 | 상태 | 액션 |
|---------|------|-----|
| Master BRD에 모든 도메인 링크가 있는가 | ✅/❌ | 없으면 추가 |
| 도메인 BRD에 Master BRD 역참조가 있는가 | ✅/❌ | 없으면 추가 |
| REQ-NNN ID가 Master BRD 전체에서 유일한가 | ✅/❌ | 중복 시 재번호 부여 |
| Core 도메인 요구사항이 최소 3개 이상인가 | ✅/⚠️ | 미달 시 경고 |
| _TBD_ 항목 수 | N개 | 사용자에게 보고 |
| Generic 도메인 처리가 명시되었는가 | ✅/❌ | Out-scope + 외부 솔루션 권고 |

### Phase 4: BRD 파일 저장

**목표**: Master BRD와 도메인별 BRD를 파일로 저장하고 다음 단계를 안내한다.

#### 4-1. Master BRD 저장

`docs/_project/BRD-master.md`로 저장한다. `docs/_project/` 디렉터리가 없으면 Bash로 `mkdir -p docs/_project`를 실행하여 생성한다.

파일 상단에 변환 메타데이터를 기록한다:
```markdown
> **변환 정보**
> - 원본 DDD 분류 문서: `{DDD 파일 경로}`
> - 변환일: {YYYY-MM-DD}
> - 서브도메인: Core {N}개 / Supporting {N}개 / Generic {N}개
> - 자동 생성 요구사항: REQ-{001}~REQ-{NNN} ({N}개)
> - 보완 필요 항목: {N}개 (_TBD_ 표시)
> - 도메인별 BRD: {링크 목록}
```

#### 4-2. 도메인별 BRD 저장

각 도메인 BRD를 `docs/{domain-kebab-case}/BRD.md`로 저장한다. 도메인명은 DDD 서브도메인명을 kebab-case로 변환한다. 해당 `docs/{domain}/` 디렉터리가 없으면 Bash로 `mkdir -p docs/{domain}`을 실행하여 생성한다.

각 도메인 BRD 상단:
```markdown
> **도메인 BRD 정보**
> - 도메인: {도메인명} ({Core/Supporting})
> - 전략: {도메인 전략 — Build/Buy/Partner 등}
> - Master BRD: [`docs/_project/BRD-master.md`](../_project/BRD-master.md)
> - 관련 Bounded Context: {BC-NN 목록}
```

#### 4-3. 저장 완료 안내

```

📋 권장 다음 단계:
1. /brd-reviewer — 생성된 BRD 품질 검토 (13섹션 완성도 분석)
```

## 도메인 BRD 출력 구조

`references/domain-brd-template.md` 템플릿을 사용한다.

각 도메인 BRD는 경량 BRD(핵심 7섹션) 포맷을 기준으로 한다:

1. 도메인 BRD 개요 (Master BRD 링크 + 도메인 컨텍스트)
2. 도메인 비즈니스 배경
3. 도메인 목표 (Master BRD 목표에서 해당 도메인 부분 추출)
4. 도메인 범위 (해당 Bounded Context)
5. 도메인 요구사항 (해당 도메인 REQ-NNN)
6. 도메인 이벤트 흐름 (DDD 섹션 7에서 해당 도메인 다이어그램 추출)
7. 도메인 용어 사전 (해당 Bounded Context 유비쿼터스 언어)

## 변환 품질 원칙

- **추적성**: 모든 REQ-NNN은 출처 DE-NN(도메인 이벤트) 또는 AG-NN(애그리게이트)을 기록
- **_TBD_ 규칙**: 자동으로 채울 수 없는 항목은 반드시 `_TBD_`로 표시 — 빈 칸보다 낫다
- **Generic 도메인 처리**: 별도 BRD를 만들지 않고 Master BRD Out-scope에 외부 솔루션 권고로 명시
- **링크 일관성**: Master BRD↔도메인 BRD 간 상호 링크를 유지
- **재실행 권장**: 설문서 작성 후 이 스킬을 재실행하여 _TBD_ 항목을 채운다
