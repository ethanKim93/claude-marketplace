# PRD 리뷰어 스킬 (다관점 리뷰)

기존 PRD 문서를 PM·엔지니어링·디자인 3개 관점에서 병렬로 평가하고, 개선을 위한 Q&A 문서를 생성합니다.
단계를 건너뛰지 말고 각 Phase를 순서대로 진행하세요.

---

## 평가 철학

좋은 PRD 리뷰는 단순히 "빠진 섹션"을 지적하는 것이 아니다. **이 문서로 팀이 실제로 일할 수 있는가**를 묻는다.

단일 관점 리뷰의 한계를 극복하기 위해 3개 서브에이전트가 **병렬로** 각자의 전문 관점에서 동시에 검토한다:

| 에이전트 | 관점 | 핵심 질문 |
|---------|------|---------|
| 🧭 **PM Agent** | 완성도·측정가능성·범위 | 이 문서로 팀이 일할 수 있는가? |
| ⚙️ **Engineering Agent** | 기술 실현가능성·NFR·테스트가능성 | 개발자가 이 문서만 보고 구현할 수 있는가? |
| 🎨 **Design Agent** | UX 흐름·접근성·에러 상태 | 사용자 관점이 충분히 반영되었는가? |

- **실행 가능성**: 엔지니어가 이 문서만 보고 티켓을 만들 수 있는가?
- **문제 중심**: 솔루션이 아닌 문제로부터 출발했는가?
- **측정 가능성**: 성공 기준이 숫자로 표현되어 있는가?
- **범위 명확성**: In-scope와 Non-goal이 모두 명시되어 있는가?
- **검증 가능성**: 인수 조건이 테스트로 전환될 수 있는가?

---

## 섹션별 평가 기준

### 표준 PRD (13섹션) 평가 기준

| 섹션 | 핵심 질문 | 결함 신호 |
|------|-----------|-----------|
| 문제 정의 | 데이터/증거가 있는가? | "사용자들이 불편해한다" (근거 없음) |
| 목표 & KPI | 숫자로 측정 가능한가? | "사용자 경험 개선" (정량화 없음) |
| 타겟 사용자 | 구체적인 페르소나가 있는가? | "모든 사용자" |
| 사용자 스토리 | As-I-So that 형식인가? | 솔루션 중심 스토리 |
| 범위 | Non-goal이 명시되었는가? | In-scope만 있고 Non-goal 없음 |
| 솔루션 | 기능 설명인가, 구현 설명인가? | 특정 기술/구현 방식을 강제하는 내용 |
| 기능 요구사항 | 우선순위가 MoSCoW로 분류되었는가? | 모두 P0, 또는 우선순위 없음 |
| 비기능 요구사항 | 수치가 포함되어 있는가? | "빠르게 동작해야 한다" |
| 인수 조건 | Given-When-Then 형식인가? | P0 요구사항에 AC 없음 |
| 의존성 | 외부 팀/시스템이 명시되었는가? | 의존성 섹션 없음 |
| 타임라인 | 마일스톤과 담당자가 있는가? | 날짜만 있고 담당자 없음 |
| 리스크 | 가능성+임팩트+대응이 있는가? | 리스크 나열만, 대응 없음 |
| 미해결 질문 | 담당자와 기한이 지정되었는가? | 질문만 있고 담당자 없음 |

### 경량 PRD (5섹션) 평가 기준

| 섹션 | 핵심 질문 |
|------|-----------|
| 문제 & 목표 | 문제-솔루션-측정이 한 단락에 있는가? |
| 솔루션 & 범위 | 포함/제외가 명확히 구분되는가? |
| P0 요구사항 | 인수 조건이 1줄이라도 있는가? |
| 의존성 & 리스크 | 주요 블로커가 식별되었는가? |
| 타임라인 | 출시 목표일과 담당자가 있는가? |

## 대화 프로세스

### Phase 0: PRD 문서 로드

**목표**: 검토할 PRD 문서를 식별하고 포맷을 감지한다.

1. 사용자가 PRD 파일 경로를 제공했는지 확인한다.
   - 경로 제공 → 해당 파일을 Read로 읽는다.
   - 경로 없음 → 사용자에게 파일 경로 또는 PRD 내용 붙여넣기를 요청한다.

2. 문서를 분석해 포맷을 감지한다:
   - 13개 이상의 섹션 → 표준 PRD
   - 5섹션 이하 → 경량 PRD
   - 보도자료 + FAQ 구조 → PR-FAQ
   - 형식 불명확 → "비정형 PRD"로 처리

3. 감지된 포맷과 문서 개요를 사용자에게 보고한다.

4. **다관점 리뷰 모드 선택**:
   - 기본값: 3개 에이전트 병렬 리뷰 (권장, 더 풍부한 피드백)
   - 빠른 검토: PM 관점 단일 리뷰 (빠른 결과)
   - 사용자가 명시하지 않으면 기본값(다관점 리뷰)으로 진행한다.

---

### Phase 1: 다관점 병렬 평가

**목표**: 3개 에이전트가 각자의 전문 관점에서 PRD를 동시에 분석한다.

#### 🧭 PM Agent — 완성도 & 측정가능성 평가

**핵심 질문**: "이 PRD로 팀이 의미 있는 결정을 내릴 수 있는가?"

평가 항목:
- 문제 정의에 데이터/증거가 있는가?
- 목표와 KPI가 숫자로 측정 가능한가?
- In-scope와 Non-goal이 모두 명시되었는가?
- 기능 요구사항에 MoSCoW 우선순위가 있는가?
- 의존성과 타임라인이 현실적인가?
- 미해결 질문에 담당자와 기한이 있는가?

#### ⚙️ Engineering Agent — 기술 실현가능성 평가

**핵심 질문**: "개발자가 이 PRD로 바로 구현을 시작할 수 있는가?"

평가 항목:
- FR이 실제로 구현 가능한 단위로 분해되었는가?
  - "FR이 실제로 2개 이상의 API 호출인가?" 감지
  - "한 FR에 여러 도메인 로직이 섞였는가?" 감지
- 비기능 요구사항(성능·보안·가용성)에 구체적인 수치가 있는가?
- 인수 조건이 자동화 테스트로 전환 가능한가?
  - Given-When-Then이 명확한가?
  - 모호한 표현("빠르게", "적절히")이 있는가?
- 외부 시스템 의존성과 API 계약이 명시되었는가?
- 데이터 모델 변경이나 마이그레이션 이슈가 암시되는가?
- 보안 고려사항 (인증, 권한, 데이터 보호)이 있는가?

#### 🎨 Design Agent — UX & 사용자 경험 평가

**핵심 질문**: "사용자 관점이 충분히 반영되었는가?"

평가 항목:
- 페르소나가 구체적인가? (나이, 기술 친숙도, 사용 맥락 포함)
- 사용자 스토리가 사용자 목표 중심인가, 아니면 기능 중심인가?
- **에러 상태와 예외 케이스**의 UX가 정의되었는가?
  - 로딩 상태, 빈 상태(Empty State), 에러 메시지
  - 네트워크 오류, 권한 없음, 데이터 없음 케이스
- 사용자 흐름(User Flow)에 진입점과 이탈점이 있는가?
- 접근성(Accessibility) 요구사항이 있는가? (WCAG 2.1)
- 모바일/반응형 대응이 언급되었는가?
- 국제화(i18n)/현지화(l10n) 고려가 필요한 기능인가?

---

### Phase 1-b: 섹션별 완성도 평가 (PM Agent 기반)

**목표**: 각 섹션을 평가 기준으로 분석하고 점수를 부여한다.

**평가 척도:**

| 등급 | 점수 | 의미 |
|------|------|------|
| 완성 | ✅ | 기준을 완전히 충족. 개선 불필요. |
| 부분 | ⚠️ | 내용은 있으나 기준 미달. 보완 필요. |
| 미흡 | ❌ | 섹션 없음 또는 내용이 매우 불충분. |

각 섹션에 대해:
1. 등급(✅ / ⚠️ / ❌) 부여
2. 구체적인 코멘트 작성 (무엇이 좋고, 무엇이 부족한지)
3. ⚠️/❌ 섹션에 대해서만 개선 방향 제시

평가 결과를 사용자에게 테이블로 보여주고 Phase 2로 진행한다.

---

### Phase 2: 다관점 평가 결과 병합

**목표**: 3개 에이전트의 발견사항을 관점별 라벨과 함께 하나의 통합 리뷰로 병합한다.

각 발견사항은 출처 에이전트를 명시한다:
- `[PM]` — 완성도·범위·우선순위 관련
- `[ENG]` — 기술 실현가능성·NFR·테스트 관련
- `[DES]` — UX 흐름·에러 상태·접근성 관련

병합 원칙:
1. 동일한 이슈를 여러 관점이 지적한 경우 → "복수 관점 공통 지적"으로 우선순위 상향
2. 관점 간 충돌하는 의견이 있으면 → 두 관점 모두 기록하고 결정을 PM에게 넘긴다
3. 각 관점의 독자적 발견은 별도 섹션에 기록

---

### Phase 2-b: 개선 질문 도출

**목표**: 3개 관점의 ⚠️/❌ 항목을 기반으로 구체적인 질문 목록을 생성한다.

**질문 유형 분류:**

| 유형 | 설명 | 예시 |
|------|------|------|
| 📋 누락 정보 | 아예 없어서 채워야 할 것 | "성공 지표의 현재 기준값은 얼마인가요?" |
| 🔍 모호한 정의 | 있지만 해석이 여러 개 가능한 것 | "'빠른 응답'이 구체적으로 몇 ms인가요?" |
| ⚡ 검증 필요 | 사실 확인이 필요한 주장이나 가정 | "월 10% 이탈률이라는 데이터의 출처는?" |
| 🔗 의존성 확인 | 다른 팀/시스템과의 연관성 | "결제 API 팀과 이미 협의가 되었나요?" |
| 🎯 범위 결정 | In-scope / Non-goal 경계 결정 | "모바일 앱도 이번 범위에 포함인가요?" |
| ⚙️ 기술 명확화 | 구현을 위한 추가 정보 필요 | "FR-003이 동기 API인가요, 비동기인가요?" |
| 🎨 UX 상태 정의 | 누락된 UI 상태 | "에러 발생 시 사용자에게 어떤 메시지를 보여주나요?" |

각 질문에는:
- 유형 레이블 + 출처 관점 `[PM]`/`[ENG]`/`[DES]`
- 관련 섹션 명시
- 질문의 중요도 (High / Medium / Low)
- 이 질문이 필요한 이유 (한 줄 설명)

---

### Phase 3: 문서 생성 및 저장

**목표**: 다관점 평가 리포트와 Q&A 문서를 파일로 저장한다.

1. 아래 두 파일을 생성한다:
   - `docs/PRD-REVIEW-{원본파일명}.md` — 다관점 통합 평가 리포트
   - `docs/PRD-QA-{원본파일명}.md` — 관점별 라벨이 붙은 Q&A 문서

2. 원본 파일명이 없는 경우 타임스탬프를 사용한다:
   - `docs/PRD-REVIEW-{YYYY-MM-DD}.md`

3. 저장 완료 후 다음 액션을 제안한다:
   - Q&A 문서를 PM·개발팀·디자이너에게 각자 해당 관점 질문 배분
   - 답변 수집 후 `/prd-writer`로 문서 업데이트
   - 팀 리뷰 미팅에서 "복수 관점 공통 지적" 항목 우선 논의

## ⚙️ Engineering 관점 평가

**핵심**: 개발자가 이 PRD로 바로 구현을 시작할 수 있는가?

| 항목 | 등급 | 발견 사항 |
|------|------|---------|
| FR 분해 적절성 | ✅/⚠️/❌ | {예: "FR-003은 실제로 API 2개 필요"} |
| NFR 수치 명확성 | ✅/⚠️/❌ | {예: "응답시간 기준값 없음"} |
| 인수조건 테스트 가능성 | ✅/⚠️/❌ | {예: "Given-When-Then 중 When 모호"} |

## 출력 원칙

- **증거 기반 평가**: "좋다/나쁘다"가 아니라 "기준의 어떤 항목을 충족/미충족했는가"로 평가한다.
- **실용적 질문**: 추상적인 질문("더 구체적으로 작성해주세요")보다 답변 가능한 질문("결제 완료율의 현재 기준값은 몇 %인가요?")을 생성한다.
- **우선순위 명확화**: 모든 것을 High로 표시하지 않는다. PRD 공유를 막는 결함만 High다.
- **긍정도 기록**: 잘 작성된 섹션은 ✅로 명시해 팀이 무엇을 잘했는지 알게 한다.
- **재검토 권장**: Q&A 답변 반영 후 다시 `/prd-reviewer`를 실행하도록 안내한다.
