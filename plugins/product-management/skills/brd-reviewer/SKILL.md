# BRD 리뷰어 스킬

기존 BRD 문서를 평가 기준에 따라 분석하고, 개선을 위한 Q&A 문서를 생성합니다.
단계를 건너뛰지 말고 각 Phase를 순서대로 진행하세요.

---

## 평가 철학

좋은 BRD 리뷰는 단순히 "빠진 섹션"을 찾는 것이 아니다. **이 문서로 경영진이 투자 결정을 내릴 수 있는가**를 묻는다.

- **비즈니스 케이스**: 왜 이 프로젝트가 필요한가? 비용 대비 가치가 명확한가?
- **의사결정 가능성**: 스폰서가 이 문서만 보고 예산과 인력을 승인할 수 있는가?
- **요구사항 추적성**: 각 요구사항이 비즈니스 목표로 역추적 가능한가?
- **측정 가능성**: 목표가 SMART 기준을 충족하여 완료 여부를 판단할 수 있는가?
- **범위 명확성**: In-Scope와 Out-of-Scope가 모두 명시되어 이해관계자 간 기대가 일치하는가?

---

## 섹션별 평가 기준

### Full BRD (13섹션) 평가 기준

| 섹션 | 핵심 질문 | 결함 신호 |
|------|-----------|-----------|
| 0. 표지 & 변경이력 | 버전·날짜·작성자·상태가 있는가? | 버전 없음, 변경이력 없음 |
| 1. 경영진 요약 | 목적·기대효과·예산·승인요청이 한 페이지에 있는가? | 너무 길거나, 숫자 없음 |
| 2. 사업 배경 | As-Is 문제가 데이터로 뒷받침되는가? | "불편하다"만 있고 수치 없음 |
| 3. 프로젝트 목표 | 각 목표가 SMART 5기준을 충족하는가? | "성능 개선", "고객 만족 향상" |
| 4. 범위 | In-Scope와 Out-of-Scope가 모두 명시되었는가? | In-Scope만 있고 Out-of-Scope 없음 |
| 5. 이해관계자 분석 | RACI 매트릭스가 작성되었는가? | 이름만 있고 역할/책임 없음 |
| 6. 비즈니스 요구사항 | REQ-NNN ID, MoSCoW, IEEE 동사가 적용되었는가? | ID 없음, 모두 Must Have, 금지어구 사용 |
| 7. 제약 & 가정 | 가정의 영향(틀렸을 때 결과)이 기록되었는가? | 가정만 나열, 영향 없음 |
| 8. 리스크 | 확률·영향·대응방안·담당자가 있는가? | 리스크 나열만, 대응 없음 |
| 9. 일정 & 마일스톤 | 마일스톤별 완료 기준과 담당자가 있는가? | 날짜만 있고 완료 기준 없음 |
| 10. 비용-편익 분석 | ROI 산식과 투자회수기간이 있는가? | 비용만 있고 편익 없음 |
| 11. 승인 | 서명란과 승인자 목록이 있는가? | 승인 섹션 없음 |
| 12. 용어집 | 도메인 용어와 약어가 정의되었는가? | 용어집 없음, 빈약함 |

### Lightweight BRD (7섹션+승인) 평가 기준

| 섹션 | 핵심 질문 |
|------|-----------|
| 표지 & 변경이력 | 버전과 날짜가 있는가? |
| 경영진 요약 | 목적과 기대효과가 2-3문장에 담겼는가? |
| 사업 배경 & 목표 | As-Is 문제와 To-Be 목표가 함께 있는가? |
| 범위 | 포함/제외가 명확히 구분되는가? |
| 이해관계자 | 주요 이해관계자와 역할이 있는가? |
| 비즈니스 요구사항 | 기능/비기능 요구사항에 우선순위가 있는가? |
| 제약 & 리스크 | 주요 제약과 리스크가 식별되었는가? |
| 승인 | 서명란이 있는가? |

## 요구사항 품질 규칙

섹션 6(비즈니스 요구사항)은 다음 규칙으로 별도 심층 평가한다.

### SMART 목표 검증 (섹션 3)
각 목표를 5가지 기준으로 확인한다:
- **S (Specific)**: 무엇을 달성하는지 명확한가? "개선"이나 "향상" 같은 모호한 동사 사용 여부 확인
- **M (Measurable)**: 숫자·비율·금액 등 측정 가능한 지표가 있는가?
- **A (Achievable)**: 달성 가능한 목표인가? 근거가 있는가?
- **R (Relevant)**: 비즈니스 문제와 직접 연결되는가?
- **T (Time-bound)**: 완료 기한이 명시되었는가?

### MoSCoW 비율 검사
- Must Have는 전체 요구사항의 **60% 이하**여야 한다
- Must Have > 60% → 일정 지연 위험 신호, Critical 이슈
- Must Have + Should Have 합산이 80% 초과 → Major 이슈
- 우선순위 표기가 없는 경우 → Major 이슈

### 요구사항 ID 체계
- `REQ-001` 형식 또는 `REQ-NNN` 형식을 사용해야 한다
- 비기능 요구사항은 유형 접두어 권장: NFR-P(성능), NFR-S(보안), NFR-A(가용성), NFR-I(통합)
- ID 없는 요구사항 → 추적성 단절, Major 이슈

### IEEE 29148 동사 규칙
- `shall` / `해야 한다` → 필수 요구사항 (Must Have)
- `should` / `하는 것이 좋다` → 권장 (Should Have)
- `may` / `할 수 있다` → 선택 (Could Have)
- **금지어구**: "가능하면", "빠르게", "사용자 친화적", "등등", "및/또는", "최대한", "필요시"

### 단일 요구사항 원칙
- 하나의 요구사항에 "그리고", "AND", "및"으로 두 가지 이상의 기능을 묶으면 안 된다
- 복합 요구사항은 테스트와 추적이 어렵다

### What vs How 검사
- 요구사항은 "무엇(What)"을 기술해야 한다, "어떻게(How)"가 아니다
- 특정 기술·플랫폼·구현 방식을 강제하는 내용이 있으면 설계 제약인지 요구사항인지 확인 필요
- 예) ❌ "Oracle DB를 사용하여 고객 데이터를 저장해야 한다" → ✅ "고객 데이터는 99.9% 가용성으로 영구 보존되어야 한다"

### 비기능 요구사항 완전성
5개 범주가 모두 다루어졌는지 확인:
- **성능**: 응답시간, 동시접속, 피크 용량
- **보안**: 인증/인가, 암호화, 컴플라이언스, 감사 로그
- **가용성**: 업타임 목표(99.x%), RTO, RPO
- **통합**: 연계 시스템, 인터페이스 유형, 오류 처리
- **데이터**: 마이그레이션, 보존 기간, 백업

## 심각도 분류 기준

BRD는 경영진 승인을 위한 게이트 문서이므로 심각도를 3단계로 구분한다.

| 심각도 | 정의 | 예시 |
|--------|------|------|
| **Critical** | BRD 승인을 차단하는 결함. 이 상태로 경영진에 제출해서는 안 된다 | SMART 목표 전무, 비용-편익 분석 없음, Must Have > 60%, 승인란 없음 |
| **Major** | 비즈니스 케이스를 약화시키는 결함. 승인은 가능하나 프로젝트 실패 위험이 높아진다 | Out-of-Scope 없음, RACI 없음, 요구사항 ID 없음, 측정 불가 목표 |
| **Minor** | 개선 기회. 문서 품질을 높이지만 승인을 차단하지는 않는다 | 용어집 빈약, 가정의 영향 미평가, 변경이력 불완전 |

## 대화 프로세스

### Phase 0: BRD 문서 로드

**목표**: 검토할 BRD 문서를 식별하고, 평가 기준 파일을 로드한 후 포맷을 감지한다.

1. 사용자가 BRD 파일 경로를 제공했는지 확인한다.
   - 경로 제공 → 해당 파일을 Read로 읽는다.
   - 경로 없음 → Glob으로 `docs/*/BRD.md` 를 탐색한다.
     - 도메인이 하나뿐이면 자동으로 해당 파일을 사용한다.
     - 도메인이 여러 개면 목록을 보여주고 사용자에게 선택을 요청한다.
     - 파일을 찾지 못하면 사용자에게 파일 경로 또는 BRD 내용 붙여넣기를 요청한다.

2. 평가 기준 참조 파일을 로드한다:
   - `./references/review-checklist.md` 를 Read한다 (섹션별 체크리스트).
   - Glob으로 `**/brd-writer/references/brd-template.md` 를 찾아 Read한다 (표준 템플릿 구조 비교용).

3. 문서를 분석해 포맷을 감지한다:
   - 섹션 0-12가 모두 있음 → Full BRD
   - 섹션 0-7 + 승인만 있음 → Lightweight BRD
   - 형식 불명확 → "비정형 BRD"로 처리 (최대한 매핑 시도)

4. 감지된 포맷과 문서 개요를 사용자에게 보고한 후 Phase 1로 진행한다.

---

### Phase 1: 섹션별 완성도 평가

**목표**: 각 섹션을 평가 기준으로 분석하고 등급을 부여하며, 요구사항 품질을 심층 분석한다.

**평가 척도:**

| 등급 | 의미 |
|------|------|
| ✅ 완성 | 기준을 완전히 충족. 개선 불필요. |
| ⚠️ 부분 | 내용은 있으나 기준 미달. 보완 필요. |
| ❌ 미흡 | 섹션 없음 또는 내용이 매우 불충분. |

**섹션별 평가 (Full BRD 기준):**

각 섹션에 대해:
1. 등급(✅/⚠️/❌) 부여
2. 구체적인 코멘트 (무엇이 좋고, 무엇이 부족한지)
3. ⚠️/❌ 섹션에 대해서만 개선 방향 제시

**요구사항 품질 심층 분석 (섹션 3, 6):**

섹션 6을 대상으로 다음을 집계한다:
- SMART 미충족 목표 수 (섹션 3에서)
- MoSCoW 비율: Must / Should / Could / Won't 각 %
- REQ-NNN ID 없는 요구사항 수
- 금지어구 발견 수와 위치 (Grep 활용)
- 복합 요구사항(AND/및) 수
- What vs How 위반 수 (구현 방식 명시된 요구사항)
- 비기능 요구사항 5범주 중 누락된 범주

평가 결과를 사용자에게 테이블로 보여주고 Phase 2로 진행한다.

---

### Phase 2: 개선 질문 도출

**목표**: ⚠️/❌ 섹션과 품질 분석 결과를 기반으로 구체적인 질문 목록을 생성한다.

**질문 유형 분류:**

| 유형 | 설명 | BRD 예시 |
|------|------|----------|
| 📋 누락 정보 | 아예 없어서 채워야 할 것 | "투자회수기간 계산 근거 데이터는?" |
| 🔍 모호한 정의 | 있지만 해석이 여러 개 가능한 것 | "'빠른 처리'가 구체적으로 몇 초인가?" |
| ⚡ 검증 필요 | 사실 확인이 필요한 주장이나 가정 | "연 30% 비용 절감 예상의 근거는?" |
| 🔗 이해관계자 확인 | 다른 팀/시스템과의 연관성 | "IT팀과 인프라 제약이 협의되었는가?" |
| 🎯 범위 결정 | In-Scope / Out-of-Scope 경계 결정 | "레거시 시스템 마이그레이션이 포함인가?" |
| 📊 정량화 필요 | 수치가 없어 측정 불가한 목표/요구사항 | "고객 만족도 목표를 수치로 표현할 수 있는가?" |

각 질문에는:
- 유형 레이블 (위 6가지 중 하나)
- 관련 섹션 명시
- 심각도 (Critical / Major / Minor)
- 이 질문이 필요한 이유 (한 줄 설명)

---

### Phase 3: 문서 생성 및 저장

**목표**: 평가 리포트와 Q&A 문서를 파일로 저장한다.

1. BRD 파일이 위치한 도메인 디렉터리를 사용한다.
   - BRD 경로가 `docs/{domain}/BRD.md` 형식이면 `{domain}`을 추출한다.
   - 도메인을 알 수 없는 경우 사용자에게 도메인명을 요청한다.

2. 아래 두 파일을 생성한다:
   - `docs/{domain}/BRD-REVIEW.md` — 평가 리포트
   - `docs/{domain}/BRD-QA.md` — Q&A 문서

3. 저장 완료 후 다음 액션을 제안한다:
   - Q&A 문서를 프로젝트 스폰서·이해관계자에게 공유
   - 답변 수집 후 `/brd-writer`로 문서 업데이트
   - Critical 이슈 해소 후 재검토 요청

## 출력 원칙

- **증거 기반 평가**: "좋다/나쁘다"가 아니라 "기준의 어떤 항목을 충족/미충족했는가"로 평가한다.
- **실용적 질문**: 추상적인 질문("더 구체적으로 작성해주세요")보다 답변 가능한 질문("고객 응답시간 목표가 현재 2.3초인데, 목표는 몇 초인가요?")을 생성한다.
- **심각도 균형**: Critical을 남발하지 않는다. 경영진 승인을 실제로 차단하는 결함만 Critical이다.
- **긍정도 기록**: 잘 작성된 섹션은 ✅로 명시해 팀이 무엇을 잘했는지 알게 한다.
- **재검토 권장**: Q&A 답변 반영 후 다시 `/brd-reviewer`를 실행하도록 안내한다.
