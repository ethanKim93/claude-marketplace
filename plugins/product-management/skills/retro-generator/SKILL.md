# 회고 생성기 스킬

TASKS.md와 프로젝트 문서를 분석하여 데이터 기반의 회고 문서를 생성합니다.
단계를 건너뛰지 말고 각 Phase를 순서대로 진행하세요.

---

## 회고 철학

좋은 회고는 주관적 감정이 아닌 **데이터**에서 시작한다. TASKS.md는 팀이 실제로 어떻게 일했는지를 기록한 가장 객관적인 증거다.

- **Keep**: 잘 된 것 — 다음에도 의도적으로 반복해야 할 것
- **Stop**: 문제가 된 것 — 다음에는 하지 않거나 다르게 해야 할 것
- **Start**: 이번엔 없었지만 도입하면 좋을 것

---

## 대화 프로세스

### Phase 0: 데이터 수집

**목표**: 회고에 필요한 소스 파일을 수집하고 회고 범위를 결정한다.

1. 파일 탐색:

   ```
   우선순위 1: 사용자가 명시한 TASKS.md 경로
   우선순위 2: docs/*/*/TASKS.md (Glob 검색 — 도메인이 하나면 자동 선택, 여러 개면 사용자에게 선택 요청)
   우선순위 3: docs/*/ROADMAP.md (Glob 검색, 일정 비교용)
   우선순위 4: docs/*/PRD.md (Glob 검색, 목표 비교용)
   ```

2. 회고 범위 확인:
   - 전체 프로젝트 회고인지, 특정 스프린트 회고인지 물어본다.
   - 스프린트 회고라면 스프린트 번호 또는 날짜 범위를 요청한다.

3. 팀원 이름 목록을 물어본다 (개인별 패턴 분석용, 선택사항).

---

### Phase 1: 정량 지표 분석

**목표**: TASKS.md에서 측정 가능한 지표를 추출한다.

#### 1-1. 완료율 분석

| 지표 | 값 |
|------|---|
| 전체 태스크 수 | {N} |
| 완료(DONE) | {N} ({%}) |
| 진행중(IN_PROGRESS) | {N} ({%}) |
| 대기(TODO) | {N} ({%}) |
| 차단됨(BLOCKED) | {N} ({%}) |

#### 1-2. BLOCKED 패턴 분석

BLOCKED 상태였던 태스크를 모두 추출하고 원인 유형을 분류:

| BLOCKED 원인 유형 | 발생 빈도 | 대표 사례 |
|----------------|---------|---------|
| 🔗 외부 의존성 (다른 팀/API 대기) | {N}건 | {예시} |
| 📋 요구사항 불명확 (PRD 재확인 필요) | {N}건 | {예시} |
| 🛠️ 기술적 장애 (환경 설정, 버그) | {N}건 | {예시} |
| 👤 리소스 부족 (담당자 없음) | {N}건 | {예시} |
| 🔄 선행 태스크 지연 | {N}건 | {예시} |

> BLOCKED 원인이 TASKS.md에 기록되어 있지 않은 경우 팀원에게 직접 물어본다.

#### 1-3. 크기 추정 정확도 분석

TASKS.md에 추정 크기(T-shirt sizing)와 실제 완료 시간이 있는 경우:

| 크기 | 예상 태스크 수 | 실제 완료된 수 | 오차 |
|------|-------------|-------------|------|
| XS | {N} | {N} | {%} |
| S | {N} | {N} | {%} |
| M | {N} | {N} | {%} |
| L | {N} | {N} | {%} |
| XL | {N} | {N} | {%} |

추정 정확도가 낮은 크기가 있으면 패턴을 기록.

#### 1-4. 스프린트 속도 (Velocity)

스프린트별 완료 태스크 수와 크기를 집계:

```
Sprint 1: DONE {N}개 (XS×{N} + S×{N} + M×{N} = 총 {N} 포인트)
Sprint 2: DONE {N}개 (... = 총 {N} 포인트)
평균 속도: {N} 포인트/스프린트
```

### Phase 2: 정성 분석

**목표**: 정량 데이터를 팀 협업 패턴으로 해석한다.

다음 질문에 대해 데이터를 근거로 답한다:

1. **가장 많이 완료된 태스크 유형은?** (design/backend/frontend/test/infra)
   - 한 유형에 집중되어 있으면 병목 또는 강점으로 해석

2. **BLOCKED 태스크 중 PRD/요구사항 불명확이 원인인 비율은?**
   - 이 비율이 높으면 다음 PRD 사이클에서 인수 조건을 더 상세히 작성해야 함

3. **크기 추정이 지속적으로 틀린 영역은?**
   - 특정 기술 영역(예: 인증, 외부 API 연동)에서 과소 추정이 반복되면 다음 로드맵에서 버퍼 증가 필요

4. **완료되지 않은 태스크가 많은 FR은?**
   - 다음 PRD 사이클에서 해당 FR의 범위·복잡도를 재검토

5. **의존성 체인에서 크리티컬 패스 지연이 발생했는가?**
   - 어떤 태스크가 가장 많은 후속 태스크를 블로킹했는가

---

### Phase 3: Keep / Stop / Start 도출

**목표**: 데이터 분석 결과를 팀이 실행할 수 있는 액션으로 변환한다.

각 항목은 반드시 **데이터 근거**와 함께 작성한다.

#### Keep (잘 된 것 — 계속)
- 근거: 어떤 지표가 이것이 효과적임을 보여주는가?
- 예: "P0 요구사항 인수 조건 사전 작성 → P0 태스크 BLOCKED 0건"

#### Stop (문제가 된 것 — 중단/변경)
- 근거: 어떤 지표가 이것이 문제임을 보여주는가?
- 예: "XL 태스크 미분해 → XL 2개 중 1개가 3스프린트 걸림 → 분해 규칙 강화 필요"

#### Start (새로 도입할 것)
- 근거: 어떤 문제를 해결하기 위한 것인가?
- 예: "외부 API 의존성 초기 식별 → BLOCKED 원인 37%가 외부 의존성 대기"

---

### Phase 4: 다음 사이클 인사이트 도출

**목표**: 다음 BRD/PRD 사이클에 직접 반영할 수 있는 구체적 개선안을 도출한다.

```
📌 다음 BRD에 반영할 사항:
- 외부 시스템 의존성을 섹션 7(제약 조건)에 더 상세히 명시
- NFR(비기능 요구사항)에 외부 API SLA 포함

📌 다음 PRD에 반영할 사항:
- P1 요구사항도 인수 조건 초안 작성 (이번에 P1 BLOCKED 비율 높음)
- XL 태스크는 로드맵 생성 전에 M 이하로 분해 강제

📌 다음 로드맵에 반영할 사항:
- 외부 API 연동 태스크는 기본 추정 × 1.5 버퍼 적용
- 인증 관련 태스크는 S 이하로 세분화
```

---

### Phase 5: 회고 문서 저장

파일명: `docs/{domain}/RETRO-{날짜 또는 스프린트명}.md`

- `{domain}`은 TASKS.md 경로(`docs/{domain}/{feature}/TASKS.md`)에서 추출한다.
- 도메인을 알 수 없는 경우 사용자에게 도메인명을 요청한다.

저장 후 다음 단계를 안내한다:

```
✅ 회고 문서가 생성되었습니다: docs/{domain}/RETRO-{파일명}.md

📋 권장 활용:
1. 팀 회고 미팅에서 이 문서를 기반으로 토론
2. Keep/Stop/Start 액션 아이템에 담당자와 기한 지정
3. 다음 BRD 작성 시 "다음 사이클 인사이트" 섹션 참고
4. /pm-pipeline으로 다음 프로젝트 사이클 시작
```

## 출력 원칙

- **데이터 우선**: 모든 Keep/Stop/Start 항목에는 수치 근거를 포함한다.
- **실행 가능성**: 추상적 관찰("커뮤니케이션 개선 필요")이 아닌 구체적 행동("PRD 리뷰 미팅을 스프린트 시작 전날로 고정")으로 작성한다.
- **블레임 없음**: 개인을 탓하는 표현 없이 프로세스와 구조에 집중한다.
- **순환 연결**: 회고 인사이트가 다음 BRD → PRD 사이클로 명시적으로 이어지도록 한다.
