# Task Runner

TASKS.md는 계획이고, 이 스킬은 실행이다.

`/roadmap-generator`가 만든 TASKS 문서에는 잘 분해된 태스크, 의존성, 인수 조건이 있다.
하지만 그건 종이 위의 계획일 뿐이다. 이 스킬은 그 계획을 실제 코드로 바꾸는 과정을 돕는다:
태스크를 골라서 → 함께 구현하고 → 완료 표시하고 → 다음으로 넘어간다.

핵심은 **TASKS.md가 single source of truth**라는 것이다.
상태 변경은 항상 이 파일에 반영되고, 진행 현황 섹션이 자동 갱신된다.

---

## 작동 모드

사용자 요청에 따라 세 가지 모드 중 하나로 진입한다:

### 모드 1: 대시보드 (진행 현황 조회)

"진행 현황 보여줘", "대시보드", "status" 같은 요청에 반응한다.

### 모드 2: 태스크 실행

"T-01-02 시작하자", "다음 태스크 진행하자" 같은 요청에 반응한다.

### 모드 3: 상태 업데이트

"T-01-02 완료", "T-03-01 블로커 있어" 같은 요청에 반응한다.

---

## TASKS.md 로딩

모든 모드의 첫 번째 단계. TASKS.md 파일을 찾아서 현재 상태를 파악한다.

1. **파일 찾기**
   - 사용자가 파일 경로를 제공했으면 Read한다.
   - 아니면 Glob으로 `docs/TASKS-*.md`를 탐색한다.
   - 여러 파일이 나오면 목록을 보여주고 선택하게 한다.

2. **상태 파싱**
   FR별 태스크 목록 테이블에서 각 태스크의 정보를 추출한다:
   - ID, 태스크명, 유형, 사이즈, 우선순위, 의존 태스크, **상태**, 설명
   - 상태 컬럼이 없는 구 형식 TASKS.md도 지원한다 (모든 태스크를 TODO로 간주).

3. **의존성 그래프 구성**
   각 태스크의 `의존 태스크` 컬럼을 파싱하여 방향 그래프를 메모리에 구성한다.
   `—`이나 빈 값은 의존성 없음을 의미한다.

4. **실행 가능 태스크 계산**
   TODO 상태이면서 모든 의존 태스크가 DONE인 태스크 = 지금 실행 가능한 태스크.
   이 목록이 이후 모든 모드에서 핵심이다.

## 모드 1: 대시보드

현재 프로젝트 진행 상황을 한눈에 보여준다.

**출력 형식:**

```
📊 프로젝트 현황: {PRD 제목}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

진행률: ████████░░░░░░░░░░░░ 40% (8/20)

| 상태 | 개수 | 비율 |
|------|------|------|
| DONE | 8 | 40% |
| IN_PROGRESS | 2 | 10% |
| TODO | 9 | 45% |
| BLOCKED | 1 | 5% |

🔄 현재 진행 중:
  - T-03-01: 검색 API 구현 (backend, M)
  - T-03-02: 검색 UI 컴포넌트 (frontend, M)

✅ 지금 실행 가능한 태스크:
  - T-02-03: 프로필 통합 테스트 (test, M) — 의존: T-02-02 ✅
  - T-04-01: 대시보드 UI 설계 (design, S) — 의존: T-01-02 ✅

🚫 차단된 태스크:
  - T-05-01: 결제 연동 (backend, L) — 사유: 외부 PG사 API 키 대기 중

📅 스프린트 진행:
  - Sprint 1: 6/6 완료 ✅
  - Sprint 2: 2/5 완료 (진행 중)
  - Sprint 3: 0/5 (대기)

💡 추천: "T-02-03 시작하자" 또는 "다음 태스크 진행"
```

대시보드 출력 후 바로 태스크 실행으로 이어갈 수 있도록 추천을 붙인다.
추천 우선순위: P0 > P1 > P2, 같은 우선순위면 크리티컬 패스에 있는 태스크 먼저.

---

## 모드 2: 태스크 실행

태스크를 선택하고, 함께 구현하고, 완료까지 진행하는 메인 루프.

### Step 1: 태스크 선택

**특정 태스크 지정 시** ("T-01-02 시작하자"):
- 해당 태스크를 찾아 의존성 충족 여부를 확인한다.
- 의존성 미충족 시:
  ```
  ⚠️ T-01-02는 아직 시작할 수 없습니다.
  미완료 의존 태스크: T-01-01 (TODO)
  T-01-01을 먼저 진행할까요?
  ```
- 이미 DONE이면: "T-01-02는 이미 완료되었습니다."
- 이미 IN_PROGRESS면: "T-01-02는 이미 진행 중입니다. 계속 진행할까요?"

**자동 추천 시** ("다음 태스크", "뭐 할 수 있어"):
- 실행 가능한 태스크 목록을 보여주고 선택하게 한다.
- 추천 순서:
  1. 크리티컬 패스에 있는 태스크 (프로젝트 일정에 가장 큰 영향)
  2. P0 > P1 > P2 우선순위
  3. 가장 많은 후행 태스크를 언블락하는 태스크
- 실행 가능한 태스크가 없으면:
  ```
  현재 실행 가능한 태스크가 없습니다.
  - IN_PROGRESS: {N}개 (완료하면 새 태스크가 언블락됩니다)
  - BLOCKED: {N}개
  진행 중인 태스크를 계속하시겠습니까?
  ```

### Step 2: 태스크 브리핑

선택한 태스크의 전체 컨텍스트를 보여준다:

```
🎯 태스크 시작: T-01-02 — 사용자 인증 API 구현

📋 상세 정보:
  - 유형: backend
  - 사이즈: M (1-3일)
  - 우선순위: P0
  - 출처: FR-01 (사용자 인증 시스템)

📎 인수 조건:
  - Given 사용자가 올바른 이메일/비밀번호를 입력했을 때
    When 로그인 API를 호출하면
    Then JWT 토큰이 반환된다
  - Given 잘못된 비밀번호를 입력했을 때
    When 로그인 API를 호출하면
    Then 401 에러가 반환된다

🔗 의존 태스크: T-01-01 (API 인터페이스 설계) ✅ DONE
🔓 완료 시 언블락: T-01-04, T-03-01

구현을 시작할까요?
```

### Step 3: TASKS.md 상태 업데이트 (IN_PROGRESS)

사용자가 시작을 확인하면:
1. TASKS.md에서 해당 태스크의 상태를 `TODO` → `IN_PROGRESS`로 변경
2. 진행 현황 섹션을 재계산하여 업데이트
3. 최종 업데이트 날짜를 갱신

### Step 4: 구현 지원

태스크 유형에 맞는 구현을 함께 진행한다.
이 단계는 대화형으로 유연하게 진행한다 — 스킬이 강제하는 것이 아니라 사용자가 원하는 방식으로 돕는다.

**유형별 가이드라인:**

- **design**: 와이어프레임 설명, API 계약서 작성, 데이터 모델 설계
- **backend**: 코드 작성, API 엔드포인트 구현, DB 마이그레이션
- **frontend**: 컴포넌트 구현, 상태 관리, API 연동
- **test**: 테스트 코드 작성, 테스트 실행, 커버리지 확인
- **infra**: CI/CD 설정, 환경 변수, 모니터링 셋업
- **docs**: API 문서 작성, README 업데이트

구현 중 수시로 인수 조건을 참조하여 올바른 방향으로 진행하고 있는지 확인한다.

### Step 5: 완료 처리

구현이 끝나면 인수 조건을 하나씩 확인한다:

```
✅ 인수 조건 검증:
  1. ✅ 올바른 이메일/비밀번호 → JWT 토큰 반환
  2. ✅ 잘못된 비밀번호 → 401 에러 반환

모든 인수 조건을 충족합니다. T-01-02를 완료 처리할까요?
```

사용자가 확인하면:
1. TASKS.md에서 상태를 `IN_PROGRESS` → `DONE`으로 변경
2. 진행 현황 섹션 재계산
3. 새로 언블락된 태스크를 알림:
   ```
   🔓 새로 실행 가능해진 태스크:
     - T-01-04: 인증 통합 테스트 (test, S)
     - T-03-01: 검색 API 구현 (backend, M)

   계속 진행하시겠습니까? 추천: T-03-01 (크리티컬 패스)
   ```

## 모드 3: 상태 업데이트

직접적인 상태 변경 요청을 처리한다.

### 완료 처리 ("T-01-02 완료", "T-01-02 done")

1. 해당 태스크의 현재 상태를 확인한다.
2. IN_PROGRESS가 아니면 확인한다: "T-01-02는 현재 {상태}입니다. 정말 완료 처리할까요?"
3. 인수 조건이 있으면 충족 여부를 물어본다.
4. TASKS.md 업데이트 (상태 → DONE, 진행 현황 재계산).
5. 언블락된 태스크 알림.

### 차단 처리 ("T-05-01 블로커", "T-05-01 blocked")

1. 상태를 `BLOCKED`로 변경한다.
2. 차단 사유를 물어본다: "어떤 이유로 차단되었나요?"
3. TASKS.md 업데이트. 해당 태스크 행 끝에 차단 사유를 기록한다.
4. 이 태스크에 의존하는 후행 태스크가 있으면 경고:
   ```
   ⚠️ T-05-01 차단으로 영향받는 태스크:
     - T-05-02 (T-05-01 의존)
     - T-06-01 (T-05-01 의존)
   이 태스크들은 T-05-01이 해소될 때까지 시작할 수 없습니다.
   ```

### 되돌리기 ("T-01-02 다시 열어줘", "T-01-02 reopen")

완료나 차단 상태를 TODO로 되돌린다.
의존 관계를 재검토하여 영향을 안내한다.

## TASKS.md 업데이트 규칙

상태를 변경할 때마다 TASKS.md에서 세 곳을 업데이트한다:

### 1. FR별 태스크 목록 테이블의 상태 컬럼

Edit 도구로 해당 행의 상태 값만 변경한다.
예: `| T-01-02 | ... | TODO | ...` → `| T-01-02 | ... | IN_PROGRESS | ...`

### 2. 진행 현황 섹션

모든 태스크의 상태를 집계하여 재계산한다:

```markdown
## 진행 현황

| 상태 | 개수 | 비율 |
|------|------|------|
| DONE | {N} | {N}% |
| IN_PROGRESS | {N} | {N}% |
| TODO | {N} | {N}% |
| BLOCKED | {N} | {N}% |

**전체 진행률**: █████████░░░░░░░░░░░ 45%

**현재 실행 가능한 태스크** (의존성 충족):
- {실행 가능한 태스크 목록}
```

진행률 바는 DONE 비율에 따라 `█`(완료)과 `░`(미완료)로 20칸을 채운다.

### 3. 메타데이터의 최종 업데이트 날짜

문서 상단의 `최종 업데이트` 필드를 오늘 날짜로 갱신한다.

### 스프린트 계획 섹션 동기화

스프린트 계획 테이블에도 상태 컬럼이 있다면 함께 업데이트한다.
FR별 태스크 목록이 정본(source of truth)이고, 스프린트 테이블은 보조 뷰이다.

## 출력 원칙

**TASKS.md가 유일한 진실**
상태는 항상 TASKS.md 파일에 저장된다. 별도 DB나 JSON 없이 마크다운 파일 하나로 관리한다.
다른 사람이 파일을 직접 열어봐도 현재 상태를 즉시 파악할 수 있어야 한다.

**의존성 존중**
의존 태스크가 DONE이 아닌 태스크는 실행할 수 없다.
예외: 사용자가 명시적으로 "의존성 무시하고 시작하자"고 하면 경고 후 허용한다.

**변경 최소화**
TASKS.md 수정 시 변경이 필요한 부분만 정확히 수정한다.
Edit 도구를 사용하여 상태 값, 진행 현황 섹션, 날짜만 업데이트한다.
태스크 설명이나 의존성 같은 다른 정보는 건드리지 않는다.

**자연스러운 루프**
태스크 완료 → 언블락 알림 → 다음 추천 → 선택 → 실행의 흐름이 끊기지 않도록 한다.
사용자가 "계속"이나 "다음"이라고만 해도 추천된 태스크로 이어갈 수 있어야 한다.

**인수 조건 기반 완료**
태스크를 완료 처리할 때 인수 조건이 있으면 반드시 확인한다.
인수 조건 없이 완료 표시하는 것도 허용하되, 인수 조건이 있는 경우 하나씩 점검을 권한다.
