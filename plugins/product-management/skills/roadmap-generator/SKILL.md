# Roadmap Generator

PRD가 완성되어도 "어디서부터 시작해야 하나"라는 질문이 남는다. 이 스킬은 그 간극을 메운다.
FR 항목들은 서로 얽혀 있다. FR-03("검색 필터")은 FR-01("검색 기능")이 없으면 의미가 없고,
FR-07("리포트")은 FR-03과 FR-05 둘 다 완료되어야 시작할 수 있다.
의존성을 무시하고 우선순위만으로 순서를 정하면 팀이 막힌다.

이 스킬은 5단계 대화형 플로우로 진행하며, 각 단계에서 사용자 확인 후 다음으로 넘어간다.
표준 13섹션 PRD와 경량 5섹션 PRD 모두 지원한다.

---

## 핵심 개념

### NOW / NEXT / LATER 프레임워크

PRD의 우선순위를 세 가지 Phase로 매핑한다:
- **NOW** (P0): 현재 스프린트. 없으면 제품이 동작하지 않는 필수 기능.
- **NEXT** (P1): 다음 1-2 스프린트. 있으면 제품이 더 좋아지는 핵심 기능.
- **LATER** (P2): 백로그. 있으면 좋지만 지금 당장은 아닌 기능.

**중요**: 의존성이 우선순위를 override한다.
P1 FR이라도 P0 FR의 선행 조건이면 NOW로 승격한다.

### 의존성 분석 원칙

세 가지 의존성 유형을 파악한다:
1. **명시적 의존성**: PRD 섹션 10(의존성 & 가정)에 직접 기술된 것
2. **데이터 흐름 의존성**: FR-B가 FR-A의 출력(API, DB 데이터)을 소비하는 경우
3. **UI 흐름 의존성**: FR-B의 사용자 흐름이 FR-A가 존재한다고 전제하는 경우

### 크리티컬 패스

가장 긴 의존성 체인 = 프로젝트 최소 기간. 이 체인의 FR 중 하나라도 지연되면
전체 프로젝트가 그만큼 지연된다. 리소스 우선 배분 대상이다.

### T-shirt 사이징

| 사이즈 | 기간 | 선택 기준 |
|--------|------|-----------|
| XS | 반나절 미만 | 설정 변경, 단순 수정, 텍스트 변경 |
| S | 반나절~1일 | 단일 컴포넌트, 패턴 명확, 불확실성 없음 |
| M | 1~3일 | 복수 컴포넌트, 약간의 불확실성 |
| L | 3~5일 | 횡단 관심사, 상당한 불확실성 |
| XL | 1주+ | 추가 분해 필요 (태스크로 부적절) |

### RICE 우선순위 점수

```
RICE = (Reach × Impact × Confidence) ÷ Effort

Reach: 분기당 영향받는 사용자 수
Impact: 0.25(최소) / 0.5(낮음) / 1(중간) / 2(높음) / 3(대규모)
Confidence: 50%(낮음) / 80%(중간) / 100%(높음)
Effort: person-days (XS=0.5, S=1, M=2, L=4, XL=8)
```

높은 RICE 점수 = 적은 노력으로 많은 사람에게 큰 임팩트.

## PRD 파싱 규칙

### 포맷 감지

표준 13섹션 PRD인지 경량 5섹션 PRD인지 판단하는 기준:
- "## 7. 기능 요구사항" 또는 `FR-\d+` 패턴이 있으면 → **표준 PRD**
- "## 요구사항 (P0만)" 또는 5개 이하의 `##` 섹션이 있으면 → **경량 PRD**

### 표준 PRD에서 추출할 정보

| PRD 섹션 | 추출 데이터 | 사용처 |
|----------|------------|--------|
| 섹션 2 (목표 & 성공 지표) | KPI 테이블 | 로드맵 성공 지표 |
| 섹션 7 (기능 요구사항) | FR ID, 설명, 우선순위, 관련 US | 로드맵 기반 데이터 |
| 섹션 8 (비기능 요구사항) | 성능/보안/접근성 항목 | infra/test 태스크 생성 |
| 섹션 9 (인수 조건) | Given-When-Then | 태스크 완료 기준 |
| 섹션 10 (의존성 & 가정) | 명시적 의존성, 외부 의존성 | 의존성 그래프 |
| 섹션 11 (타임라인) | 마일스톤, 날짜 | Gantt 차트 앵커 |
| 섹션 12 (리스크 & 대응) | 리스크, 확률, 영향도 | 리스크 버퍼 계산 |

### 경량 PRD 처리

경량 PRD에는 FR ID가 없다. 요구사항 항목에 FR-01, FR-02...를 자동 부여하고,
의존성은 텍스트 의미 분석으로만 추론한다.
Phase 1에서 사용자에게 이 사실을 알리고 의존성을 반드시 확인받는다.

## Phase 0: PRD 로딩 및 분석

**목표**: PRD 파일을 로드하고 구조를 파악하여 이후 분석의 재료를 준비한다.

1. **PRD 파일 찾기**
   - 사용자가 파일 경로를 제공했으면 그 파일을 Read한다.
   - 도메인명만 제공했으면 `docs/{domain}/PRD.md` 경로를 직접 시도한다.
   - 도메인을 특정하지 않았으면 Glob으로 `docs/*/PRD.md` 패턴을 탐색한다.
   - 여러 파일이 나오면 목록을 보여주고 어떤 PRD를 사용할지 선택하게 한다.
   - `docs/` 디렉터리가 없으면 Glob으로 `**/*.md`에서 PRD 패턴을 탐색한다.
   - PRD 파일 경로에서 도메인명을 추출한다: `docs/{domain}/PRD.md` → `{domain}`

2. **포맷 감지 및 데이터 추출**
   - 위의 파싱 규칙에 따라 표준/경량 PRD를 판별한다.
   - FR 목록을 추출한다 (ID, 설명, 우선순위).
   - 비기능 요구사항을 추출한다 (infra/test 태스크 생성에 필요).
   - 섹션 10의 명시적 의존성을 추출한다.
   - 섹션 11의 마일스톤과 날짜를 추출한다 (없으면 Phase 2에서 사용자에게 시작일 질문).
   - 섹션 12의 리스크 목록을 추출한다.

3. **기능 챕터 파싱**
   섹션 7에서 `### 7.N Feature: {name}` 패턴을 탐지한다:
   - 패턴이 **존재하면** → 기능 챕터 모드: 각 챕터별로 FR 목록을 분리 추출하고, Feature 이름을 kebab-case로 변환하여 `{feature-key}`로 저장한다.
     - 예: `### 7.1 Feature: User Login` → `{feature-key}` = `user-login`
     - 각 챕터에 속한 FR ID 목록을 매핑 테이블로 구성한다.
   - 패턴이 **없으면** → 단일 기능 모드: 전체 FR을 하나의 기능 그룹으로 처리하고 `{feature-key}` = `default`로 설정한다.

4. **Phase 0 요약 제시**
   ```
   ✅ PRD 로딩 완료: {PRD 파일 경로}
   - 도메인: {domain}
   - 포맷: 표준 13섹션 / 경량 5섹션
   - FR 수: P0({N}개) / P1({N}개) / P2({N}개)
   - 기능 챕터: {N}개 발견 / 없음 (단일 기능 모드)
     {챕터가 있을 경우: 각 챕터명과 FR 수 나열}
   - 명시적 의존성: {N}개 발견
   - 마일스톤: {N}개 / 타임라인 정보 {있음/없음}
   - 리스크 항목: {N}개
   ```
   이후: "Phase 1(의존성 분석)으로 진행할까요?"

## Phase 1: 의존성 분석

**목표**: FR 간 의존성을 분석하고, 크리티컬 패스를 식별하며, 사용자의 확인을 받는다.
의존성이 잘못 설정되면 로드맵과 태스크 전체가 틀어지므로 이 단계의 사용자 확인이 가장 중요하다.

1. **의존성 도출**
   - 명시적 의존성(섹션 10)을 그래프의 시작점으로 삼는다.
   - 각 FR을 검토하여 데이터/UI 흐름 의존성을 추론한다:
     - "FR-B의 설명이 FR-A의 결과물(API 응답, DB 레코드, UI 화면)을 언급하는가?"
     - "FR-B의 사용자 흐름이 FR-A가 이미 존재한다고 전제하는가?"
   - 추론된 의존성은 "추론됨"으로 표시하여 명시적 의존성과 구분한다.

2. **의존성 테이블 제시**
   ```
   | FR ID | 요구사항 | 우선순위 | 선행 FR | 의존성 근거 |
   |-------|----------|---------|---------|------------|
   | FR-01 | {설명}   | P0      | —       | —          |
   | FR-03 | {설명}   | P0      | FR-01   | 명시적     |
   | FR-04 | {설명}   | P1      | FR-01   | 추론됨 (UI 흐름) |
   ```

3. **Mermaid 의존성 플로우차트 생성**
   크리티컬 패스를 빨간색(`classDef critical`)으로 강조한다.
   ```mermaid
   graph TD
       FR01["FR-01: {설명}"] --> FR03["FR-03: {설명}"]
       classDef critical fill:#ff6b6b,stroke:#c0392b,color:#fff
       class FR01,FR03 critical
   ```

4. **크리티컬 패스 계산**
   모든 가능한 경로 중 가장 긴 체인을 찾아 명시한다.
   ```
   크리티컬 패스: FR-01 → FR-03 → FR-06
   예상 최소 기간: {N}주 (이 체인을 순차 완료하는 데 걸리는 시간)
   ```

5. **의존성-우선순위 충돌 경고**
   P1/P2 FR이 P0 FR의 선행 조건인 경우 명시적으로 알린다:
   ```
   ⚠️ 우선순위 충돌: FR-04(P1)는 FR-01(P0)의 선행 조건입니다.
      → Phase 2에서 FR-04를 NOW로 승격 예정
   ```

6. **사용자 확인**
   "의존성 분석 결과입니다. 수정하거나 추가할 의존성이 있나요?
   확인되면 Phase 2(로드맵 구성)로 진행합니다."

## Phase 2: 로드맵 구성

**목표**: 의존성이 반영된 NOW/NEXT/LATER 페이즈를 구성하고 Gantt 차트와 함께 제시한다.

1. **초기 Phase 배정**: P0 → NOW, P1 → NEXT, P2 → LATER

2. **의존성 기반 승격**
   크리티컬 패스 분석 결과에 따라 우선순위 Override를 적용한다.
   예: P1이지만 P0의 선행 조건 → NOW로 승격. 이 변경은 명시적으로 표시한다.

3. **위상 정렬(Topological Sort)**
   각 Phase 내에서 FR을 의존성 순서에 맞게 정렬한다.
   선행 FR 없는 것부터 시작하여 의존성이 해소된 순서로 나열한다.

4. **타임라인 추정**
   - PRD 섹션 11의 마일스톤이 있으면 그것을 앵커로 삼는다.
   - 없으면: "로드맵 시작일이 언제인가요?" 질문한다.
   - 각 FR의 예상 기간은 FR 복잡도(P0 평균 1-2주, P1 평균 1주)와 비기능 요구사항 부하를 고려한다.
   - 리스크 버퍼: 섹션 12의 높음(High) 리스크에 +30%, 중간(Medium)에 +15% 추가.

5. **Phase 계획 테이블 제시**
   ```
   Phase NOW: {시작일} ~ {종료일}
   | FR ID | 요구사항 | 예상 기간 | 리스크 |
   ...
   ```

6. **Mermaid Gantt 차트 생성**
   크리티컬 패스 FR은 `crit` 태그를 붙인다.
   각 FR 완료 시점에 마일스톤을 배치한다.

7. **리스크 조정 타임라인 테이블**
   기본 기간, 리스크 요인, 버퍼, 조정 기간을 보여준다.

8. **사용자 확인**
   "로드맵 초안입니다. 기간이나 Phase 배정을 조정하시겠습니까?
   확인되면 Phase 3(태스크 분해)로 진행합니다."

## Phase 3: 태스크 분해

**목표**: 각 FR을 실행 가능한 개발 태스크로 분해하고 RICE 점수와 스프린트 배분을 제안한다.

1. **FR별 태스크 생성**
   각 FR에 대해 해당 FR의 특성에 따라 필요한 태스크 유형을 선별하여 생성한다:
   - **design**: UI 설계가 필요한 FR에만 (프론트엔드 없는 순수 백엔드 FR은 생략)
   - **backend**: API, DB 스키마, 비즈니스 로직이 있는 FR
   - **frontend**: 사용자 인터페이스가 있는 FR
   - **test**: 모든 P0 FR (P1 이상도 인수 조건이 있으면 포함)
   - **infra**: PRD 비기능 요구사항에서 도출 (배포, 모니터링 등)
   - **docs**: P0 완료 후 API 문서가 필요한 경우

   태스크 ID 체계: `T-{FR번호}-{순번}` (예: FR-01의 두 번째 태스크 → T-01-02)

2. **인수 조건 연결**
   PRD 섹션 9의 Given-When-Then을 각 태스크의 완료 기준으로 할당한다.
   P0 태스크는 반드시 인수 조건이 있어야 한다.

3. **RICE 점수 계산**
   각 태스크에 대해 PRD의 사용자 스토리(Reach)와 비즈니스 목표(Impact)를 참고하여
   Reach, Impact, Confidence, Effort를 추정하고 RICE 점수를 계산한다.

4. **스프린트 배분 제안**
   2주 스프린트 기준으로 태스크를 배분한다:
   - Sprint 1: 의존성이 없는 P0 태스크
   - Sprint 2: Sprint 1에 의존하는 P0 태스크
   - Sprint 3+: P1 태스크 (의존성 순서 준수)

5. **XL 태스크 경고**
   XL 사이즈로 추정되는 태스크는 추가 분해를 제안한다:
   ```
   ⚠️ T-02-02는 XL 사이즈로 추정됩니다.
      더 작은 단위로 분해하는 것을 권장합니다.
   ```

6. **기능 챕터별 태스크 그룹화**
   기능 챕터 모드인 경우 각 태스크를 해당 FR이 속한 기능 챕터에 매핑한다:
   - `{feature-key}` → [T-{FR번호}-{순번}, ...] 형태의 그룹 목록 구성
   - 기능 챕터를 넘나드는 의존성(cross-feature dependency)은 별도로 표시한다.
   - 단일 기능 모드에서는 그룹화 없이 전체 태스크를 단일 목록으로 유지한다.

7. **태스크 요약 제시 및 사용자 확인**
   ```
   총 {N}개 태스크 생성:
   - P0: {N}개 ({N} person-days 예상)
   - P1: {N}개 ({N} person-days 예상)
   - P2: {N}개 ({N} person-days 예상)

   기능 챕터 모드인 경우:
   - {feature-key}: {N}개 태스크
   - {feature-key}: {N}개 태스크
   ...
   ```
   "태스크 분해 결과입니다. 추가하거나 수정할 태스크가 있나요?
   확인되면 Phase 4(문서 생성)로 진행합니다."

## Phase 4: 문서 생성

**목표**: 확인된 내용을 기반으로 도메인 구조에 맞춰 문서를 생성하고 다음 단계를 안내한다.

1. **출력 경로 결정**
   Phase 0에서 추출한 `{domain}`과 Phase 3에서 구성한 `{feature-key}` 목록을 기반으로 경로를 결정한다:
   - 로드맵 파일: `docs/{domain}/ROADMAP.md` (도메인 레벨, 항상 단일 파일)
   - 태스크 파일 (기능 챕터 모드): 기능별 N개 파일
     - `docs/{domain}/{feature-key}/TASKS.md`
     - 예: `docs/user-auth/login/TASKS.md`, `docs/user-auth/profile/TASKS.md`
   - 태스크 파일 (단일 기능 모드): 1개 파일
     - `docs/{domain}/tasks/default/TASKS.md`

2. **로드맵 문서 생성** (`docs/{domain}/ROADMAP.md`)
   `references/roadmap-template.md`의 구조를 따라 작성한다.
   Phase 0-3에서 분석한 모든 내용을 채워넣는다.
   Mermaid 다이어그램(의존성 플로우차트, Gantt 차트) 모두 포함한다.
   기능 챕터 모드인 경우 다음 항목을 ROADMAP.md에 추가한다:
   - 기능별 섹션 분리 (각 Feature의 FR 목록, 의존성, 태스크 수)
   - 기능 간 cross-feature 의존성 표시
   - 각 기능의 TASKS.md 파일 경로 링크 목록:
     ```
     ## 기능별 태스크 파일
     - [{feature-key}](../{feature-key}/TASKS.md): {N}개 태스크
     ```

3. **태스크 문서 생성**
   `references/task-template.md`의 구조를 따라 작성한다.

   - **기능 챕터 모드**: 각 `{feature-key}`별로 독립적인 TASKS.md를 생성한다.
     - 해당 기능에 속한 FR에서 파생된 태스크만 포함한다.
     - 다른 기능의 FR에 대한 cross-feature 의존성은 참조 링크로 표시한다:
       ```
       선행 조건: [T-01-01](../../{other-feature}/TASKS.md#T-01-01) (cross-feature)
       ```
     - 각 파일 상단에 소속 정보를 명시한다:
       ```
       도메인: {domain}
       기능: {feature-key}
       ROADMAP 참조: ../ROADMAP.md
       ```
   - **단일 기능 모드**: FR별 태스크 목록, 의존성 매트릭스, RICE 테이블, 스프린트 계획을 모두 포함한다.

4. **완료 요약 제시**

   기능 챕터 모드인 경우:
   ```
   ✅ 문서 생성 완료

   📄 docs/{domain}/ROADMAP.md
      - Phase NOW: {N}개 FR ({시작일}~{종료일})
      - Phase NEXT: {N}개 FR
      - Phase LATER: {N}개 FR
      - 크리티컬 패스: {FR-01 → FR-03 → ...}
      - 총 예상 기간: {N}주 (리스크 버퍼 포함)
      - 기능 {N}개 포함

   📋 기능별 TASKS.md ({N}개 파일):
      - docs/{domain}/{feature-key-1}/TASKS.md ({N}개 태스크, {N} person-days)
      - docs/{domain}/{feature-key-2}/TASKS.md ({N}개 태스크, {N} person-days)
      ...
   ```

   단일 기능 모드인 경우:
   ```
   ✅ 문서 생성 완료

   📄 docs/{domain}/ROADMAP.md
      - Phase NOW: {N}개 FR ({시작일}~{종료일})
      - Phase NEXT: {N}개 FR
      - Phase LATER: {N}개 FR
      - 크리티컬 패스: {FR-01 → FR-03 → ...}
      - 총 예상 기간: {N}주 (리스크 버퍼 포함)

   📋 docs/{domain}/tasks/default/TASKS.md
      - 총 {N}개 태스크 ({N} person-days)
      - {N} 스프린트 분량 (2주 기준)
   ```

   공통 다음 단계:
   ```
   다음 단계:
   1. 팀 리뷰 미팅에서 ROADMAP 문서를 공유하세요
   2. 기능별 TASKS 문서로 담당자를 배정하세요
   3. Planning Poker로 T-shirt 사이징을 검증하세요
   4. Jira/Linear에 Phase NOW 태스크를 임포트하세요
   ```

## 출력 원칙

**의존성이 우선순위를 이긴다**
우선순위만으로 빌드 순서를 결정하지 않는다. P1 FR이라도 P0 FR의 블로커라면 먼저 구현해야 한다.
이 override는 항상 명시적으로 표시한다.

**Mermaid 정확성**
생성하는 모든 Mermaid 다이어그램은 문법적으로 유효해야 한다:
- 노드 ID에 공백 사용 금지 (FR-01 → FR01)
- 특수문자나 공백이 있는 레이블은 반드시 큰따옴표로 감싼다
- `graph TD`는 의존성 플로우차트, `gantt`는 타임라인에 사용한다

**태스크 독립성**
각 태스크는 한 명이 맡아 1 스프린트(2주) 내에 완료할 수 있어야 한다.
XL 사이즈 태스크는 사용자에게 분해를 권장한다.

**PRD 추적성**
모든 태스크에는 출처 FR이 명시된다. 로드맵의 모든 항목은 PRD의 FR 또는 비기능 요구사항으로 역추적 가능해야 한다.

**단계별 확인**
Phase 1, 2, 3 각각에서 반드시 사용자 확인을 받은 후 다음 단계로 넘어간다.
확인 없이 Phase를 건너뛰지 않는다.
