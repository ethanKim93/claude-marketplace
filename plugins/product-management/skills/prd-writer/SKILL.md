# PRD 작성 스킬

PRD 베스트 프랙티스를 바탕으로 사용자와 대화하며 실행 가능한 제품 요구사항 문서를 만듭니다.
단계를 건너뛰지 말고 사용자 확인을 받으며 진행하세요.

---

## PRD 포맷 선택 기준

| 포맷 | 적합한 상황 | 분량 |
|------|-------------|------|
| **표준 PRD** | 신규 제품, 대규모 기능, 팀 간 협업 필요 | 13섹션 |
| **경량 PRD** | 기존 기능 개선, 소규모 변경, 긴급 배포 | 5섹션 |
| **PR-FAQ** | 비전 중심 제품, 경영진 설득, 장기 로드맵 | 보도자료 + FAQ |

---

## 핵심 개념 참조

### 좋은 PRD의 조건

**실행 가능성** — 엔지니어가 PRD를 읽고 바로 티켓을 만들 수 있어야 한다. 요구사항마다 인수 조건이 있고, 우선순위가 명확하며, 모호한 부분이 없어야 한다.

**문제 중심** — "무엇을 만들 것인가"보다 "왜 필요한가"로 시작한다. 문제가 명확하면 솔루션은 자연스럽게 따라온다.

**범위의 명확성** — 이번 이터레이션에서 하는 것과 하지 않는 것을 모두 명시한다. 비범위(non-goal)가 없으면 스코프 크리프가 생긴다.

**측정 가능한 성공** — "사용자 경험 개선"은 목표가 아니다. "태스크 완료 시간 20% 단축"이 목표다.

### 요구사항 우선순위 (MoSCoW / P-tier)

| 레벨 | 의미 | 기준 |
|------|------|------|
| P0 (Must) | 이것 없으면 출시 불가 | 핵심 비즈니스 요구사항, 법적 의무 |
| P1 (Should) | 강력히 권장, 대부분 포함 | 핵심 사용자 가치, 경쟁 패리티 |
| P2 (Could) | 여력이 있으면 포함 | 편의 기능, 향후 개선 후보 |
| P3 (Won't) | 이번엔 하지 않음 | 비범위로 명시 |

### 인수 조건 작성법 (Given-When-Then)

```
Given [특정 상황/전제 조건]
When [사용자가 어떤 행동을 하면]
Then [시스템이 어떻게 반응해야 한다]
```

## 대화 프로세스

### Phase 0: 컨텍스트 수집

**목표**: 프로젝트 현황을 파악하고 적합한 PRD 포맷을 추천한다.

1. 사용자가 프로젝트 경로를 제공하면 다음 파일을 읽어 컨텍스트를 확보한다:
   - `CLAUDE.md` / `README.md` — 프로젝트 개요, 기술 스택
   - `package.json` / `build.gradle` — 사용 기술 확인
   - `docs/` 폴더 — 기존 문서화 수준 확인

2. 사용자에게 질문:
   - "어떤 유형인가요?" → 신규 제품 / 신규 기능 / 기능 개선 / 버그 수정
   - "예상 규모는?" → 대형(팀 전체) / 중형(2~3명) / 소형(혼자)

3. 답변을 바탕으로 포맷을 추천하고 확인을 받은 후 다음 Phase로 진행한다.

---

### Phase 1: 문제 정의 & 목표

**목표**: 왜 이 기능이 필요한지, 성공을 어떻게 측정할지 정의한다.

**질문할 내용:**
- "어떤 문제를 해결하나요? 이 문제가 존재한다는 증거(데이터, 지원 티켓, 사용자 피드백)가 있나요?"
- "이 기능이 없으면 어떤 일이 생기나요? 비즈니스 임팩트는?"
- "성공을 어떻게 측정하시겠습니까?"

**성공 지표 예시 (참고용으로 제시):**
- 전환율: "결제 완료율 5% 향상"
- 효율성: "태스크 완료 시간 30% 단축"
- 참여도: "DAU 15% 증가"
- 오류 감소: "에러 레이트 50% 감소"

답변을 정리해 보여주고 확인받은 후 다음 Phase로 이동한다.

---

### Phase 2: 타겟 사용자 & 사용자 스토리

**목표**: 누가 이 기능을 쓸 것인지, 어떤 방식으로 쓸 것인지 정의한다.

**질문할 내용:**
- "주요 사용자는 누구인가요? 2~3개의 페르소나를 알려주세요."
  - 페르소나 예시: "30대 직장인, 모바일 주 사용, 기술 친숙도 중간"
- "각 페르소나가 이 기능으로 달성하고 싶은 것은?"

**사용자 스토리 작성:**
사용자 답변을 바탕으로 스토리를 초안 작성하고 검토를 요청한다.

```
As a [페르소나],
I want to [특정 행동을 하고 싶다],
So that [원하는 결과를 얻을 수 있다].
```

---

### Phase 3: 범위 & 솔루션

**목표**: 무엇을 만들고 무엇은 만들지 않을지 명확히 한다.

**질문할 내용:**
- "이번 이터레이션(MVP)에 반드시 포함되어야 하는 것은?"
- "의도적으로 제외하는 것은? 이유는?" (비범위 명시)
- "솔루션 방향은 있으신가요, 아니면 제안이 필요한가요?"

비범위는 반드시 명시적으로 문서화한다 — 없으면 나중에 "이것도 해줘"가 된다.

---

### Phase 4: 요구사항 & 인수 조건

**목표**: Phase 2~3의 사용자 스토리를 구체적인 요구사항으로 전환하고 우선순위를 부여한다.

1. 각 사용자 스토리를 기능 요구사항(FR)으로 분해한다
2. P0/P1/P2 우선순위를 부여한다
3. **P0 요구사항에는 반드시** Given-When-Then 형식의 인수 조건을 작성한다

**비기능 요구사항도 확인:**
- 성능: "응답 시간 200ms 이하"
- 보안: "인증된 사용자만 접근"
- 접근성: "WCAG 2.1 AA 준수"
- 확장성: "동시 사용자 1만 명 지원"

---

### Phase 5: 의존성, 타임라인, 리스크

**목표**: 구현을 가로막을 수 있는 요소들을 미리 식별한다.

**질문할 내용:**
- "다른 팀이나 외부 시스템에 의존하는 것이 있나요?"
- "기술적으로 불확실한 부분이 있나요? 스파이크(PoC)가 필요한 부분은?"
- "타임라인 제약이 있나요? (출시 데드라인, 의존하는 다른 기능)"

리스크는 가능성과 임팩트를 함께 평가하고 대응 방안을 제시한다.

---

### Phase 6: 문서 생성 & 품질 검증

**목표**: 수집한 정보를 선택한 포맷의 PRD로 컴파일하고 파일로 저장한다.

1. 선택된 포맷의 템플릿으로 PRD를 작성한다
2. 품질 체크리스트를 실행한다
3. `docs/PRD-{kebab-case-title}.md`로 저장한다
4. 다음 단계를 제안한다 (팀 리뷰, 티켓 생성 등)

## 품질 체크리스트

PRD를 완성하기 전 아래 항목을 검토하세요:

- [ ] 모든 P0 요구사항에 인수 조건이 있는가?
- [ ] 성공 지표가 측정 가능한 숫자로 표현되었는가?
- [ ] 비범위(Non-goals)가 명시적으로 적혀 있는가?
- [ ] 엔지니어가 이 문서만 보고 티켓을 만들 수 있는가?
- [ ] 미해결 질문이 모두 기록되고 담당자가 지정되었는가?
- [ ] 의존성이 모두 식별되었는가?

## 출력 원칙

- **단계별 진행**: Phase를 나눠 사용자 확인 후 다음 단계로 이동. 한 번에 모든 섹션을 채우지 않는다.
- **질문 우선**: 모호한 부분은 추측하지 말고 물어본다. 특히 성공 지표와 범위는 반드시 사용자가 직접 정의해야 한다.
- **문제에서 솔루션으로**: 솔루션을 먼저 제안하지 말고, 문제와 목표가 명확해진 후 솔루션 섹션을 채운다.
- **티켓 친화적 출력**: 각 요구사항은 독립적인 Jira/Linear 티켓으로 만들 수 있어야 한다.
- **비범위 강조**: 사용자가 비범위를 말하지 않으면 반드시 물어본다.
