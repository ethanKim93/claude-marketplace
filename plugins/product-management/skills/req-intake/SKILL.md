# Requirements Intake — 요구사항 수집·정리 스킬

어떤 형태로 들어오든 상관없이 원자재(raw input)를 받아 **BRD-writer가 바로 소비할 수 있는 표준 형식**으로 변환한다.
출력 파일(`REQUIREMENTS-INTAKE.md`)은 `/brd-writer`의 직접 입력으로 사용되도록 설계된다.

---

## 1. 실행 흐름

### 1단계: 입력 소스 파악

사용자가 파일 경로나 텍스트를 직접 붙여넣거나 "@파일" 으로 지정한 경우 즉시 처리한다.  
아무것도 제공되지 않았으면 다음을 질문한다:

```
요구사항 원본이 어디에 있나요?
1. 현재 디렉터리 파일 (경로 알려주세요)
2. 지금 여기에 직접 붙여넣기
3. 현재 디렉터리를 스캔해서 요구사항 파일 자동 탐색
```

### 2단계: 원본 읽기

- 파일 경로가 주어진 경우 `Read`로 읽는다.
- 자동 탐색 요청 시 `Glob`으로 다음 패턴을 순서대로 스캔한다:
  ```
  **/*requirement*.md, **/*요구사항*.md, **/*needs*.md,
  **/*spec*.md, **/*기획*.md, **/*회의록*.md,
  **/*.txt, **/*feedback*.md, **/*issue*.md
  ```
  발견된 파일 목록을 보여주고 사용자에게 대상을 선택받는다.
- 텍스트를 직접 붙여넣은 경우 그대로 처리한다.

### 3단계: 도메인명 확인

`Glob`으로 `docs/*/BRD.md` 패턴을 스캔한다.
- 기존 도메인이 있으면 목록을 보여주고 연결할 도메인을 선택받는다.
- 없으면 사용자에게 도메인명을 묻는다 (예: `payment`, `user-management`).

### 4단계: 원본 분석 — 항목 추출

원본 텍스트를 분석하여 모든 항목을 추출하고 **4가지 유형**으로 분류한다:

| 유형 코드 | 유형 | 키워드 예시 |
|-----------|------|------------|
| `FEAT` | 기능 요구사항 | "~해야 한다", "~기능 추가", "~가 필요하다", "개발 요청" |
| `NFR` | 비기능 요구사항 | "성능", "보안", "가용성", "응답시간", "동시접속", "암호화" |
| `BUG` | 버그·결함 | "오류", "안 된다", "깨진다", "버그", "오작동", "에러" |
| `IMPR` | 개선 요청 | "불편하다", "개선", "UX", "속도 느림", "사용성" |

**판단이 애매한 경우** → `FEAT`로 분류하고 후처리 시 사용자에게 확인받는다.

### 5단계: 항목 정규화

각 추출 항목을 다음 속성으로 정규화한다:

```
ID       : RI-NNN (Running Number, 001부터 순번)
유형     : FEAT / NFR / BUG / IMPR
제목     : 30자 이내 한 줄 요약
원문     : 원본에서 발췌한 텍스트 (최대한 원문 보존)
출처     : 파일명 또는 "직접 입력" + 줄번호/섹션
도메인   : 연관 기능 영역 (예: 인증, 결제, 알림, 관리자)
우선순위 : Must / Should / Could / Unknown
비고     : 모호하거나 추가 확인이 필요한 사항
```

**우선순위 판단 기준**:
- "반드시", "필수", "법적", "규정", "오픈 불가" → `Must`
- "중요", "필요", "해야" → `Should`
- "있으면 좋겠다", "가능하면", "차후" → `Could`
- 단서 없음 → `Unknown`

### 6단계: 중복·충돌 감지

- **중복**: 동일한 의미의 항목이 2개 이상이면 하나로 병합하고 원문 출처를 모두 기록한다.
- **충돌**: 서로 상반된 요구사항(예: "로그인 불필요" vs "로그인 필수")을 감지하면 `⚠️ CONFLICT` 태그를 붙인다.

### 7단계: REQUIREMENTS-INTAKE.md 생성

`docs/{domain}/REQUIREMENTS-INTAKE.md`로 저장한다.

## 3. 분석 품질 기준

출력을 생성하기 전에 다음을 자체 검증한다:

- [ ] 원본의 모든 문단·항목이 최소 1개 이상의 RI-NNN에 매핑되었는가
- [ ] 유형 분류가 일관성 있는가 (FEAT에 버그가 섞이지 않았는가)
- [ ] 제목이 30자 이내이고 동사로 끝나는가 (예: "로그인 중복 세션 차단")
- [ ] 원문이 삭제·축약되지 않고 보존되었는가
- [ ] `Unknown` 우선순위 항목에 대해 확인 질문이 작성되었는가
- [ ] 충돌 탐지가 수행되었는가

## 4. 지원 입력 형식 예시

| 입력 형태 | 처리 방식 |
|-----------|-----------|
| 마크다운 파일 (.md) | 헤더·리스트·표를 구조 단위로 파싱 |
| 일반 텍스트 (.txt) | 줄바꿈·빈 줄을 단락 구분자로 사용 |
| 카카오톡·슬랙 대화 | 발화자별로 그룹핑 후 요구 발화 추출 |
| 이메일 본문 | 제목+본문을 함께 분석, 수신자 정보 출처로 기록 |
| 회의록 | Action Item·결정사항 섹션 우선 처리 |
| Jira/Linear 덤프 | 티켓 ID를 출처로, 제목+설명을 항목으로 처리 |
| 엑셀 복붙 | 탭/쉼표 구분 행을 개별 항목으로 처리 |
| 구두 설명 (텍스트 요약) | 문장 단위로 분해하여 항목 추출 |

## 5. 실행 완료 후 안내 메시지

파일 저장 후 다음을 출력한다:

```
✅ REQUIREMENTS-INTAKE.md 생성 완료
   경로: docs/{domain}/REQUIREMENTS-INTAKE.md
   총 {N}건 추출 — FEAT {n} / NFR {n} / BUG {n} / IMPR {n}
   확인 필요: {n}건 | 충돌: {n}건

다음 단계:
   → 확인 필요 항목({n}건)을 검토한 뒤
   → /brd-writer 를 실행하여 BRD 초안을 자동 생성하세요.
```
