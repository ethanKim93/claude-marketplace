# BRD Writer — 비즈니스 요구사항 문서 작성 스킬

BRD(Business Requirements Document)는 프로젝트 착수 단계에서 작성하는 공식 문서로, 조직이 **"왜(Why)"** 이 프로젝트를 하고 **"무엇(What)"** 을 달성해야 하는지를 정의한다. 개발 방법론(Agile/Waterfall)에 상관없이 사용되며, 경영진과 이해관계자가 예산과 범위를 승인하는 근거가 된다.

---

## 1. BRD 작성 프로세스

사용자가 BRD 작성을 요청하면 다음 순서로 진행한다:

### 1단계: 맥락 파악 (질문 목록)

작성을 시작하기 전에 모르는 항목을 사용자에게 질문한다. 한 번에 너무 많이 묻지 말고 핵심 사항만 먼저 물어본다:

1. **프로젝트 목적**: 이 프로젝트를 하는 이유, 해결하려는 문제
2. **주요 이해관계자**: 스폰서(예산 승인자), 사업 부서, 최종 사용자
3. **예상 범위**: 새 시스템 구축인지, 기존 시스템 개선인지, 연동 대상은?
4. **제약 조건**: 예산 한도, 오픈 기한, 기술 제약(레거시 시스템 등)
5. **작성 목적**: 내부 승인용인지, 외주 발주용인지, 투자 심의용인지

정보가 불충분하면 합리적인 가정(Assumption)을 명시하고 진행한다.

### 2단계: 문서 구조 선택

상황에 따라 적절한 구조를 선택한다:

- **Full BRD** (투자 심의, 대규모 프로젝트): 아래 전체 섹션
- **Lightweight BRD** (내부 팀용, 소규모): 섹션 1–7 + 승인
- **IT BRD** (소프트웨어 개발): Full + 비기능 요구사항 섹션 강화

### 3단계: 문서 작성

`references/brd-template.md`의 섹션별 작성 지침을 참고하여 작성한다.

## 2. BRD 핵심 섹션 구성

### 필수 섹션 (모든 BRD)

| # | 섹션명 | 핵심 질문 |
|---|--------|----------|
| 0 | 표지 & 변경 이력 | 누가, 언제, 어떤 버전? |
| 1 | 경영진 요약 | 1페이지로 요약하면? |
| 2 | 비즈니스 배경 & 필요성 | 왜 지금 이 프로젝트인가? |
| 3 | 프로젝트 목표 | SMART 기준으로 무엇을 달성하나? |
| 4 | 프로젝트 범위 | 무엇을 하고, 무엇을 하지 않나? |
| 5 | 이해관계자 분석 | 누가 관여하고 각자의 역할은? |
| 6 | 비즈니스 요구사항 | 시스템이 무엇을 해야 하나? |
| 7 | 제약 조건 & 가정 | 어떤 제한이 있고 무엇을 전제하나? |
| 8 | 리스크 & 완화 방안 | 잘못되면 어떤 일이 생기나? |
| 9 | 일정 & 마일스톤 | 언제까지 무엇을 완료하나? |
| 10 | 비용-편익 분석 | 투자 대비 효과는? |
| 11 | 승인 | 누가 서명하나? |
| 12 | 용어 사전 | 도메인 용어와 약어 정의 |

> **참고**: 경영진 요약(섹션 1)은 문서를 완성한 후 마지막에 작성한다.

## 3. 요구사항 작성 핵심 원칙

### SMART 목표 기준

| 기준 | 나쁜 예 | 좋은 예 |
|------|---------|---------|
| Specific | "성능 개선" | "API 응답시간 50% 단축" |
| Measurable | "빠르게" | "95%의 요청을 2초 이내 응답" |
| Achievable | "100% 자동화" | "수동 작업 80% 감소" |
| Relevant | (비즈니스 목표 연결 없음) | "운영 비용 절감으로 수익성 개선" |
| Time-bound | "빨리" | "2026년 Q3까지" |

### MoSCoW 우선순위

| 등급 | 의미 | 권고 비율 |
|------|------|-----------|
| **M**ust Have | 없으면 프로젝트 실패 | **최대 60%** |
| **S**hould Have | 중요하지만 없어도 출시 가능 | 약 20% |
| **C**ould Have | 있으면 좋은 추가 기능 | 약 15% |
| **W**on't Have | 이번 버전 제외 | 약 5% |

> Must Have가 60%를 초과하면 일정 지연 위험이 매우 높아진다.

### 요구사항 ID 체계

```
[REQ-NNN] 시스템은 [조건]에서 [주체]가 [행위]를 [결과]까지 [기준]으로 수행할 수 있어야 한다.

예시:
[REQ-001] 시스템은 로그인된 사용자가 검사 요청서를 제출했을 때,
          30초 이내에 요청 번호를 발급하고 이메일 알림을 발송해야 한다. (Must Have)
```

**동사 선택 규칙 (IEEE 29148 기반)**:
- `해야 한다 / shall` → 필수 요구사항
- `하는 것이 좋다 / should` → 권고 사항
- `할 수 있다 / may` → 선택 사항
- 금지 표현: "가능하면", "빠르게", "사용자 친화적", "etc.", "및/또는"

### 단일 요구사항 원칙

```
나쁨: "시스템은 인증을 처리하고 보고서를 생성하며 이메일을 발송해야 한다"

좋음:
[REQ-001] 시스템은 사용자 인증을 처리해야 한다
[REQ-002] 시스템은 보고서를 생성해야 한다
[REQ-003] 시스템은 이메일을 발송해야 한다
```

## 4. IT 프로젝트 비기능 요구사항 체크리스트

소프트웨어 개발 BRD에는 다음 항목을 반드시 포함한다:

```
성능 (Performance)
  [ ] 응답시간: ___ 초 이내 (95th percentile)
  [ ] 동시 사용자: ___ 명
  [ ] 피크 타임 처리 능력

보안 (Security)
  [ ] 인증 방식: (JWT / SAML / OAuth)
  [ ] 권한 모델: (RBAC / ABAC)
  [ ] 데이터 암호화 기준
  [ ] 규정 준수: (GDPR / 개인정보보호법)
  [ ] 감사 로그 보존 기간

가용성 (Availability)
  [ ] 목표 업타임: ___% (99.9% = 연간 8.7시간 허용 다운)
  [ ] RTO (목표 복구 시간): ___ 시간
  [ ] RPO (목표 복구 시점): ___ 시간

통합 (Integration)
  [ ] 연동 시스템 목록
  [ ] 인터페이스 유형: (REST API / Batch / File / Event)
  [ ] 에러 처리 정책

데이터 (Data)
  [ ] 마이그레이션 범위
  [ ] 보존 기간
  [ ] 백업 주기
  [ ] 마스터 데이터 출처 시스템
```

## 5. 흔한 실수 & 품질 점검

BRD를 완성한 후 다음을 반드시 검토한다:

**내용 품질**
- [ ] 모든 목표가 SMART 기준을 충족하는가
- [ ] 각 요구사항에 고유 ID가 있는가
- [ ] Must Have ≤ 60% 인가
- [ ] 기능/비기능 요구사항이 분리되었는가
- [ ] "빠르게", "사용자 친화적" 같은 모호한 표현이 없는가
- [ ] AND로 묶인 복합 요구사항이 없는가
- [ ] **솔루션(How)이 아닌 요구사항(What)만 기술**되었는가

**범위 & 구조**
- [ ] In-scope와 Out-of-scope가 모두 명시되었는가
- [ ] 가정 사항이 문서화되었는가
- [ ] 승인 서명란이 있는가
- [ ] 변경 이력 테이블이 있는가

**핵심 실수 10가지**:
1. Must Have 60% 초과 → 일정 지연 확정
2. 솔루션을 요구사항으로 착각 ("Oracle DB를 써야 한다" → BRD에 금지)
3. 이해관계자 일부 누락 → 나중에 범위 분쟁
4. 측정 불가능한 목표 ("개선", "향상", "최적화")
5. Out-of-scope 미명시 → 범위 크리프(scope creep)
6. 가정(Assumption) 미문서화
7. 버전 관리 부재
8. 개발 시작 후 작성 (착수 전에 완료해야 함)
9. 요구사항 ID 미부여 → 추적 불가
10. 승인 프로세스 생략

## 6. BRD vs PRD vs FRD — 언제 무엇을 쓰나

| 구분 | BRD | PRD | FRD |
|------|-----|-----|-----|
| 핵심 질문 | **Why** — 왜 하는가 | **What** — 무엇을 만드는가 | **How** — 어떻게 구현하는가 |
| 작성 시점 | 프로젝트 착수 전 | BRD 승인 후 | PRD 확정 후 |
| 주 독자 | 경영진, 스폰서, 사업부서 | PM, 디자이너, 개발팀 | 개발자, QA, 아키텍트 |
| 상세 수준 | 전략적·추상적 | 기능 수준 | 기술 세부 사항 |

> 경영진 예산 승인이 필요하다 → **BRD 먼저**
> 팀이 구성되고 무엇을 만들지 정의한다 → **PRD**
> 개발자가 스펙대로 구현한다 → **FRD**
