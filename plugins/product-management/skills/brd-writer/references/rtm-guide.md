# 요구사항 추적성 매트릭스 (RTM: Requirements Traceability Matrix)

BRD가 승인된 후 RTM을 생성하여 전체 개발 수명주기에서 요구사항을 추적한다.

---

## RTM의 목적

- 모든 비즈니스 요구사항이 설계·구현·테스트되었는지 검증
- 변경 요청(Change Request) 시 영향 범위 신속 파악
- 테스트 커버리지 보장
- 감사(Audit) 대응 근거 자료

---

## RTM 기본 구조

```markdown
| 요구사항 ID | 요구사항 설명 | 출처 (BRD §) | 설계 문서 | 개발 Task | 테스트 케이스 | 상태 |
|------------|-------------|-------------|---------|---------|-------------|------|
| REQ-001 | 사용자 로그인 | BRD §6.1 | FRD-D01 | TASK-101 | TC-001, TC-002 | 완료 |
| REQ-002 | 역할별 권한 제어 | BRD §6.1 | FRD-D02 | TASK-102 | TC-003, TC-004 | 개발중 |
| REQ-003 | 계정 잠금 처리 | BRD §6.1 | FRD-D03 | TASK-103 | TC-005 | 설계중 |
| NFR-P001 | 응답시간 2초 이내 | BRD §6.2 | NFR-D01 | — | PTC-001 | 미착수 |
```

---

## 상태 코드

| 상태 | 의미 |
|------|------|
| 미착수 | 아직 설계/개발 시작 전 |
| 설계중 | FRD 작성 단계 |
| 개발중 | 구현 진행 중 |
| 테스트중 | QA/UAT 진행 중 |
| 완료 | 테스트 통과, 인수 완료 |
| 보류 | 변경 검토 중 |
| 삭제 | 요구사항 제거 (사유 기록 필수) |

---

## RTM 업데이트 시점

1. **BRD 승인 후**: RTM 초안 생성 (요구사항 ID + 설명만)
2. **FRD 완성 후**: 설계 문서 컬럼 채우기
3. **스프린트/개발 Task 생성 후**: 개발 Task 컬럼 채우기
4. **테스트 케이스 작성 후**: 테스트 케이스 컬럼 채우기
5. **테스트 완료 후**: 상태 업데이트

---

## 변경 요청 처리 절차

요구사항 변경이 발생하면:

1. **영향 분석**: RTM에서 해당 REQ-ID를 찾아 연결된 설계/개발/테스트 항목 확인
2. **변경 범위 산정**: 영향받는 항목 목록화
3. **CCB(변경통제위원회) 승인**: 범위/비용/일정 변경 승인
4. **문서 업데이트**: BRD 버전 업 + RTM 업데이트
5. **팀 공지**: 관련 이해관계자에게 변경 내용 통보
