## 섹션별 평가

| 섹션 | 등급 | 코멘트 |
|------|------|--------|
| 0. 표지 & 변경이력 | ✅ | 버전 1.2, 작성일, 작성자, 문서 상태(Review) 명시. 변경이력 테이블에 v1.0~v1.2 3개 항목 기록. 검토자·승인자는 표지에 별도 미기재되나 섹션 11 승인란에서 확인 가능. |
| 1. 경영진 요약 | ✅ | 프로젝트 목적, 기대효과(채용 기간 45일→20일, 처리 건수 8→15건), 예산 4억원, 구축 기간 9개월 명시. 승인 요청 항목 포함. 단일 단락으로 간결하게 요약됨. |
| 2. 사업 배경 | ✅ | As-Is 현황을 수치(평균 45일, 월 24건, 이탈률 30%)로 뒷받침. To-Be 목표 기술. 미추진 시 리스크(우수 인재 이탈) 함의. 다만 미추진 시 리스크를 독립 항목으로 명시하면 더 완성도가 높아짐. |
| 3. 프로젝트 목표 | ✅ | G-001~G-003 모두 SMART 5기준 충족: 목표ID·설명·지표·기준값·목표값·기한 테이블로 구조화됨. 3개 목표가 사업 배경(처리 시간, 생산성, 이탈률)과 일대일 대응함. |
| 4. 범위 | ✅ | In-Scope 7개 항목, Out-of-Scope 4개 항목 모두 명시. Out-of-Scope 항목에 제외 사유(2단계 추진, 추후 검토)가 기재됨. 의존성·전제조건도 SAP 연동으로 간접 명시. |
| 5. 이해관계자 분석 | ✅ | 이해관계자 4명(이름, 조직, 역할, 관심사, 영향력, 참여 수준) 기록. RACI 매트릭스 5개 활동×5개 역할로 완성. 공백 없이 R/A/C/I가 적절히 배분됨. |
| 6. 비즈니스 요구사항 | ⚠️ | REQ-ID 10개, NFR-ID 7개 모두 존재. MoSCoW 분포 Must 60%/Should 20%/Could 20%로 기준(60% 이하) 경계선. NFR은 성능·보안·가용성·통합 4범주 존재하나 데이터(NFR-D) 범주 누락. REQ-004에 복합 요구사항(공고 등록 + 메일 발송) 혼재. NFR-S001에 구현 방식(AES-256) 명시로 What vs How 경계 모호. |
| 7. 제약 & 가정 | ⚠️ | 제약사항 테이블 4개 항목 기록. 가정 테이블 3개 항목 중 첫 번째 가정("SAP HR 시스템 API 연동이 기술적으로 가능하다")의 '틀렸을 때 영향' 컬럼이 비어 있음. 나머지 2개 가정은 영향이 기재됨. |
| 8. 리스크 | ✅ | R-001~R-003 모두 ID·설명·발생가능성·영향도·우선순위·대응방안·담당자 구비. 대응방안이 PoC 수행, 보안 컨설팅, 버퍼 확보 등 구체적으로 기술됨. |
| 9. 일정 & 마일스톤 | ✅ | M1~M5 마일스톤 모두 완료기준·목표일·담당자·의존성 포함. 완료기준이 "서명 완료", "단위 테스트 95%", "Critical 버그 0건"으로 측정 가능하게 작성됨. |
| 10. 비용-편익 분석 | ✅ | 초기 비용 4억원, 연간 운영 비용 1,500만원 항목별 기재. 편익 테이블(4,500만원/년) 및 ROI 산식(≈15%), 투자회수기간(7년) 명시. 다만 투자회수기간 7년은 장기로 경영진 질의 가능성 있음 — 편익 추정 근거 보강 권장. |
| 11. 승인 | ✅ | 승인 요청 문구와 서명 테이블(역할·이름·서명·날짜) 4명 구성. 미서명 상태이나 구조적으로 완전함. |
| 12. 용어집 | ⚠️ | UAT, ROI, FTE 3개 항목만 정의됨. 본문에 등장하는 RTO, RPO, RACI, MoSCoW, SAP, AES-256, PoC, BRD, NFR, FTE 외 도메인 핵심 용어(예: 서류 심사, 합격 통보 자동화, 이력서 마이그레이션 기준 등) 미정의. |

## 요구사항 품질 분석

| 항목 | 결과 | 판정 |
|------|------|------|
| SMART 미충족 목표 | 0개 (G-001~G-003 모두 5/5 충족) | ✅ |
| Must Have 비율 | 60% (기능 6/10 + NFR Must 포함 시 더 높아질 수 있음) | ⚠️ 경계선 |
| REQ-ID 없는 요구사항 | 0개 (REQ-001~010, NFR-P001~S002, A001~002, I001 모두 부여) | ✅ |
| 금지어구 발견 | 0건 | ✅ |
| 복합 요구사항(AND) | 1개 (REQ-004: 면접 일정 등록 + 자동 안내 메일 발송 혼재) | ⚠️ |
| What vs How 위반 | 1개 (NFR-S001: "AES-256으로 암호화"로 구현 방식 명시) | ⚠️ |
| 비기능 요구사항 누락 범주 | 데이터(NFR-D) 범주 누락: 마이그레이션 범위, 보존기간, 백업 주기 미정의 | ⚠️ |

### MoSCoW 상세 분포 (기능 요구사항 10개 기준)

| 구분 | 개수 | 비율 |
|------|------|------|
| Must Have | 6 | 60% |
| Should Have | 2 | 20% |
| Could Have | 2 | 20% |
| Won't Have | 0 | 0% |

> Must Have 60%는 허용 기준(60% 이하) 정확히 경계에 위치. 향후 요구사항 추가 시 비율이 초과될 위험 있음. NFR 7개 중 Must Have가 5개(71%)로 비기능 요구사항 쪽 비율이 높은 점도 함께 모니터링 권장.

## 핵심 보완 사항

> BRD가 경영진에 제출되기 전에 반드시 해결해야 할 항목들

### Critical
*해당 없음. 현재 BRD에는 승인을 차단하는 Critical 결함이 존재하지 않습니다.*

### Major
1. **NFR 데이터 범주 누락**: In-Scope에 "기존 채용 데이터 마이그레이션 (최근 2년)"이 포함되어 있으나 NFR-D(데이터) 요구사항이 전무함. 마이그레이션 범위, 데이터 보존 기간, 백업 주기, 마스터 데이터 원천이 미정의되면 개발·운영 단계에서 분쟁 소지가 있음.
2. **REQ-004 복합 요구사항 분리 필요**: "면접 일정을 등록하고 지원자에게 자동 안내 메일을 보낼 수 있어야 한다"는 두 기능(일정 등록 + 메일 발송)을 하나의 요구사항으로 묶음. 테스트 추적성 및 개발 범위 분리를 위해 두 REQ ID로 분리 권장.

### Minor
1. **SAP 연동 가정의 영향 미기재**: 가정 테이블 첫 번째 항목("SAP HR 시스템 API 연동이 기술적으로 가능하다")의 '틀렸을 때 영향' 컬럼이 비어 있음. 이 가정은 R-001 리스크와도 연결되는 고영향 가정이므로, 영향(예: SAP 연동 불가 시 수동 동기화 또는 배치 연동으로 전환, 추가 비용 ×천만원, 일정 X주 지연)을 반드시 기재해야 함.
2. **용어집 빈약**: 3개 항목(UAT·ROI·FTE)만 정의. 본문 전반에서 사용되는 RTO, RPO, RACI, MoSCoW, SAP, AES-256, PoC, NFR 등 주요 약어와 "서류 심사", "합격 통보 자동화" 같은 도메인 용어가 미정의. 외부 이해관계자(경영진, 법무팀 등)가 문서를 이해하는 데 장벽이 될 수 있음.
3. **NFR-S001 What vs How 경계**: "AES-256으로 암호화하여 저장"은 구현 방식을 명시하는 설계 제약에 해당. "개인정보는 업계 표준 암호화 수준으로 저장되어야 하며 관련 법령(개인정보보호법, GDPR)을 충족해야 한다"로 재작성하거나, 별도 설계 제약 섹션으로 이동 권장.
4. **투자회수기간 7년 근거 보강**: ROI 15%, 회수기간 7년은 내부 승인 단계에서 "너무 길다"는 질의가 나올 수 있음. 비용 절감 추정(2,000만원/년)의 산출 근거와 10년 운영 가정의 근거를 보충하면 경영진 신뢰도 향상.
