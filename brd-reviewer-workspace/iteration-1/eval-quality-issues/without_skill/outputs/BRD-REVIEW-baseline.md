## 1. 총평

이 BRD는 13개 표준 섹션을 모두 포함하고 있으며, 경영진 요약부터 용어집까지 구조적 완성도는 높습니다. 그러나 **요구사항 품질**과 **MoSCoW 우선순위 배분**, **ROI 계산 결과**에 심각한 문제가 발견되어, 현 상태로는 경영진 승인을 받기 어렵습니다. 아래 항목들을 반드시 수정한 후 재검토를 권장합니다.

## 3. 주요 문제 및 개선 권고

### 3.1 요구사항 품질 문제 (심각)

#### 문제 1 — 구현 기술 강제 지정 (REQ-001)

> "시스템은 **Oracle DB를 사용하여** 모든 재무 데이터를 저장해야 하고 자동 백업 기능도 제공해야 한다"

BRD는 **비즈니스 요구사항** 문서입니다. 특정 기술 솔루션(Oracle DB)을 명시하는 것은 BRD의 역할을 벗어나며, 솔루션 선정 자유도를 불필요하게 제한합니다. 해당 결정은 기술 설계 단계(SRD/SDD)에서 다루어야 합니다.

**추가 문제:** REQ-001은 두 가지 요구사항(데이터 저장 + 자동 백업)을 하나의 항목에 묶고 있습니다. 요구사항은 단일 책임 원칙에 따라 분리해야 합니다.

**권고 수정:**
- REQ-001a: "시스템은 모든 재무 데이터를 안정적으로 저장하고 무결성을 보장해야 한다"
- REQ-001b: "시스템은 정기적인 자동 백업을 제공하고 복구 시점(RPO)은 [X시간] 이내여야 한다"

---

#### 문제 2 — 모호한 성능 표현 (REQ-002)

> "사용자는 **가능하면 빠르게** 월별 손익계산서를 조회할 수 있어야 한다"

"가능하면 빠르게"는 측정 불가능한 표현입니다. 이는 개발자, 테스터, 발주자가 서로 다른 기준을 갖게 하여 납품 검수 분쟁의 원인이 됩니다. 비기능 요구사항(NFR-P001)에는 "3초 이내"라는 구체적 기준이 있음에도 기능 요구사항에는 모호한 표현이 사용된 점이 일관성 문제를 일으킵니다.

**권고 수정:**
- "사용자는 월별 손익계산서를 **[N]초 이내**에 조회할 수 있어야 한다 (최대 조회 기간: 직전 24개월)"

---

#### 문제 3 — 모호한 기능 범위 (REQ-005, REQ-006)

- REQ-005: "**다양한 형식**의 재무 보고서" — '다양한'이 몇 종류인지, 어떤 형식(PDF, Excel, 화면 조회 등)인지 불명확합니다.
- REQ-006: "**사용자 친화적인** 대시보드" — '사용자 친화적'은 주관적 표현으로 검수 기준이 될 수 없습니다.

**권고 수정:**
- REQ-005: "시스템은 손익계산서, 재무상태표, 현금흐름표를 PDF 및 Excel 형식으로 출력할 수 있어야 한다"
- REQ-006: "대시보드는 재무 KPI [항목 나열], 재고 현황, 구매 현황을 한 화면에서 제공해야 한다"

### 3.2 MoSCoW 우선순위 배분 문제 (중요)

전체 요구사항(기능 9개 + 비기능 2개 = 11개) 중 Must Have가 9개로 **82%**에 달합니다. (기능 요구사항만으로는 7/9 = **78%**)

MoSCoW 방법론의 일반 권고 기준:
- Must Have: **60% 이하**
- Should Have + Could Have: 40% 이상

**Must Have 과다 배분의 위험:**
1. 모든 것이 필수이면 우선순위의 의미가 없어집니다.
2. 예산 또는 일정 압박 시 실질적인 범위 조정이 불가능해집니다.
3. 개발팀이 "어디서부터 시작할 것인가"를 판단하기 어렵습니다.

**권고:** Must Have는 **프로젝트 목표 달성에 직결되고, 없으면 시스템이 작동 불가한 요구사항**으로 한정하세요. REQ-005(보고서 형식), REQ-006(대시보드), REQ-007(마이그레이션) 등의 우선순위 재검토를 권장합니다.

### 3.3 ROI 음수 문제 (긴급)

현재 문서의 ROI 계산 결과는 **약 -8%**입니다.

```
ROI = (1.3억 × 10년 - 15억 - 8천만 × 10년) / (15억 + 8천만 × 10년) × 100
    = (13억 - 15억 - 8억) / (15억 + 8억) × 100
    = -10억 / 23억 × 100
    ≈ -43.5%
```

> 계산식 자체도 오류가 있습니다. 문서상 "약 -8%"는 계산 결과와 맞지 않습니다. 실제로는 약 -43%입니다.

**ROI 음수의 의미:** 이 프로젝트는 10년 기준으로도 투자비용을 회수하지 못합니다. 경영진 승인을 요청하는 BRD에 ROI 음수를 그대로 기재하면 승인 거부 가능성이 매우 높습니다.

**권고 대응 방안:**
1. **편익 항목 확대 검토:** 문서에 정량화되지 않은 편익(장애 감소로 인한 매출 손실 방지, 컴플라이언스 리스크 감소, 사용자 생산성 향상 등)을 발굴하고 정량화하세요.
2. **비용 구조 재검토:** 인프라 클라우드 전환 시 초기 비용 감소 시나리오를 검토하세요.
3. **비재무적 편익 명시:** 정량화가 어렵더라도 규제 대응, 리스크 감소, 인재 유치 등의 전략적 가치를 명시적으로 기술하세요.
4. **ROI 계산 오류 수정:** 현재 "-8%"라고 기재된 수치와 실제 계산 결과가 불일치합니다. 계산식을 재검토하세요.

### 3.4 범위와 요구사항의 불일치

- **REQ-009:** "모바일 브라우저에서도 주요 기능에 접근할 수 있어야 한다" — **Out-of-Scope에 '모바일 앱'이 명시**되어 있습니다. 모바일 브라우저 접근은 모바일 앱과 다르나, 독자에게 혼선을 줄 수 있습니다. 범위 항목에서 모바일 웹 접근 가능 여부를 명시적으로 언급하거나, REQ-009에 "모바일 앱 제외, 반응형 웹 기반"임을 명확히 하세요.

### 3.5 리스크 항목 보완 필요

현재 리스크 3개는 적습니다. 다음 리스크가 누락되어 있습니다:

| 누락 리스크 | 이유 |
|------------|------|
| 벤더 종속(Vendor Lock-in) | 특정 ERP 솔루션 도입 시 전환 비용 발생 위험 |
| 예산 초과 | R-003(일정 지연)과 별도로 비용 관리 리스크 필요 |
| 보안/개인정보 위반 | 재무 데이터 클라우드 이전 시 컴플라이언스 리스크 |
| 핵심 인력 이탈 | 프로젝트 기간 중 주요 담당자 변동 위험 |

## 4. 긍정적 요소

- **As-Is/To-Be 구조:** 사업 배경이 수치로 명확히 기술되어 있어 문제 상황 이해가 용이합니다.
- **SMART 목표:** 프로젝트 목표가 측정 가능한 지표와 기한으로 정의되어 있습니다.
- **RACI 매트릭스:** 이해관계자별 책임이 구체적으로 정의되어 있습니다.
- **마일스톤 완료 기준:** 각 마일스톤에 완료 조건이 명시되어 있어 진도 관리가 가능합니다.
- **용어집:** 주요 약어를 정의하여 독자 이해를 돕습니다.
