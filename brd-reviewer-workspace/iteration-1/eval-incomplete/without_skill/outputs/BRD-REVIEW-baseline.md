## 섹션별 상세 검토

### 1. 경영진 요약

**현재 상태:** 2문장 수준의 단락으로 예산(3억원)과 기간(6개월)만 언급.

**문제점:**
- 이 프로젝트가 왜 지금 필요한지 전략적 맥락이 없음
- 기대 ROI 또는 비용 절감 규모 등 경영진이 판단할 수 있는 수치 부재
- 승인 요청 항목(예산, 인력, 타임라인)이 명확히 정리되어 있지 않음
- 핵심 리스크 한 줄 요약조차 없음

**개선 방향:** 문제 → 해결책 → 기대 효과 → 요청 사항 순으로 1페이지 이내 재작성 필요.

---

### 2. 사업 배경

**현재 상태:** 현황 수치(전화 문의 70%, 대기 15분, 불만 월 200건)가 포함되어 있어 배경 근거는 상대적으로 양호.

**문제점:**
- 수치의 출처(조사 기간, 데이터 소스) 불명확
- 경쟁사 벤치마크 또는 업계 평균 대비 현황 분석 없음
- '지금 하지 않으면 어떻게 되는가(현상 유지 리스크)' 서술 없음
- 고객 불만의 구체적 유형(배송, 환불, 기술 문의 등) 분류 없음

**개선 방향:** 데이터 출처 명시, 현상 유지 시 연간 손실 추정액 추가.

---

### 3. 프로젝트 목표

**현재 상태:** 3개의 불릿 포인트로 나열. SMART 기준 미충족.

**문제점:**
- **Specific 미흡:** "향상", "절감", "개선"이라는 모호한 동사만 사용
- **Measurable 부재:** 수치 목표 전혀 없음 (예: "전화 문의 30% 감소")
- **Time-bound 미흡:** 언제까지 달성할지 시점 없음
- 목표 3개가 서로 독립적인지, 우선순위가 있는지 알 수 없음

**개선 방향 예시:**
```
- 6개월 내 전화 문의 비율 70% → 40%로 감소
- 월 운영 비용 15% 절감 (연간 약 X천만원)
- 고객 만족도(CSAT) 현재 대비 10점 이상 향상 (목표 80점)
```

---

### 4. 범위 정의

**현재 상태:** In-Scope 4개 항목만 정의.

**문제점:**
- **Out-of-Scope 섹션 완전 누락:** 모바일 앱, ERP 연동, 다국어 지원 등이 포함되는지 알 수 없음
- 레거시 시스템 연동 여부 불명확
- 가정(Assumptions) 항목 없음
- 제약사항(Constraints) 항목 없음 — 예: 기술 스택 제한, 보안 정책, 규정 준수(GDPR 등)

**개선 방향:** Out-of-Scope 명시, 주요 시스템 연동 목록, 가정·제약 사항 별도 테이블 추가.

---

### 5. 이해관계자 분석

**현재 상태:** 3명의 이름/조직/역할만 기재된 단순 테이블.

**문제점:**
- 영향도(High/Med/Low)·관심도 매트릭스 없음
- 각 이해관계자의 승인 권한(Approver / Reviewer / Informed) 미구분
- 연락처 또는 소통 채널 없음
- 최종 의사결정권자(Executive Sponsor) 명확히 지정되지 않음
- 외부 이해관계자(고객, 규제 기관 등) 고려 없음

**개선 방향:** RACI 매트릭스 또는 영향도·관심도 2×2 매트릭스 추가.

---

### 6. 비즈니스 요구사항

#### 6-1. 기능 요구사항

**현재 상태:** REQ-001 ~ REQ-005, 5개 항목. MoSCoW 우선순위 부분 적용.

**문제점:**
- 모든 항목이 Must Have — Should Have, Could Have 항목 없음 (우선순위 차별화 부재)
- 인증/보안 요구사항 없음 (로그인 방식, 세션 만료, 2FA 등)
- 비밀번호 찾기, 회원가입, 탈퇴 등 기본 계정 관리 요구사항 없음
- 불만 처리 이후 워크플로우(상태 변경, 답변, 종결) 요구사항 없음
- 데이터 보존 기간, 개인정보 처리 요구사항 없음

#### 6-2. 비기능 요구사항

**현재 상태:** 3개의 불릿 포인트, 모두 측정 불가능한 수식어.

**문제점:**
- "빠르게 동작" → 구체적 응답 시간 목표 없음 (예: 95th percentile 3초 이내)
- "보안이 강화" → 어떤 보안 수준인지 불명확 (OWASP Top 10? ISO 27001? 암호화 기준?)
- "사용자 친화적" → 측정 기준 없음 (예: 사용성 테스트 성공률 80% 이상)
- 가용성(SLA), 확장성, 재해복구(RTO/RPO) 요구사항 전혀 없음
- 접근성(WCAG 등) 요구사항 없음

**개선 방향 예시:**
```
| 항목 | 요구사항 | 측정 기준 |
|------|----------|-----------|
| 성능 | 페이지 로딩 | 95% 요청 3초 이내 |
| 가용성 | 서비스 업타임 | 월 99.5% 이상 |
| 보안 | 인증 | JWT + 세션 2시간 만료 |
| 확장성 | 동시 접속 | 최대 500 CCU 지원 |
```

## 누락 섹션 목록

아래 섹션은 일반적인 BRD 표준(BABOK, 기업 내부 BRD 가이드라인)에서 필수로 요구되나 본 문서에 완전히 없음.

| 섹션 | 중요도 | 설명 |
|------|--------|------|
| 재무 분석 / ROI | 높음 | 예산 3억원의 근거, 기대 절감액, 투자회수 기간 |
| 리스크 분석 | 높음 | 주요 리스크, 발생 가능성, 영향도, 완화 방안 |
| 일정 / 마일스톤 | 높음 | 6개월 계획의 단계별 일정 및 산출물 |
| 성공 기준 (KPI) | 높음 | 프로젝트 완료 시 성공 판단 기준 |
| 의존성 | 중간 | 다른 프로젝트, 시스템, 외부 벤더 의존성 |
| 데이터 요구사항 | 중간 | 데이터 이관, 통합, 보존 정책 |
| 법적·컴플라이언스 요구사항 | 중간 | 개인정보보호법, 전자상거래법 등 적용 법규 |
| 승인 서명 | 낮음 | 이해관계자 최종 승인 란 |

## 전체 개선 우선순위

1. **즉시 수정 (Blocker):** 프로젝트 목표에 SMART 수치 목표 추가
2. **즉시 수정 (Blocker):** Out-of-Scope 섹션 추가
3. **즉시 수정 (Blocker):** 재무 분석(ROI) 섹션 추가
4. **단기 보완:** 비기능 요구사항 측정 가능하게 재작성
5. **단기 보완:** 리스크 분석 섹션 추가
6. **단기 보완:** 일정/마일스톤 섹션 추가
7. **중기 보완:** 이해관계자 RACI 매트릭스 완성
8. **중기 보완:** 법적·컴플라이언스 요구사항 검토
